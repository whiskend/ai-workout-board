# 운동 기록 인증 게시판 기능 명세서

> 이 문서는 `운동기록기획.md` 다음 단계 문서입니다.  
> 목적은 서비스 아이디어를 실제 과제 구현 단위로 쪼개고, 어떤 기능을 MVP로 만들지 정하는 것입니다.

## 0. 프로젝트 한 줄 정의

사용자가 운동 기록을 게시글로 남기면, AI가 이전 기록과 비교해 점진적 과부하 관점에서 다음 운동 목표를 추천해주는 운동 인증 게시판입니다.

## 1. 기술 스택 결정

이번 과제는 완성 가능성과 AI 기능 구현 난이도를 기준으로 다음 스택을 사용합니다.

| 영역 | 선택 |
| --- | --- |
| Frontend | React + TypeScript |
| Backend | NestJS + TypeScript |
| AI Server | FastAPI + Python |
| Database | PostgreSQL + pgvector |
| Auth | JWT |
| ORM | NestJS는 Prisma, FastAPI는 SQLAlchemy 또는 raw SQL |
| LLM | GPT-4.1 mini 이상 |
| RAG | PostgreSQL/pgvector 기반 운동 기록 검색 |
| MCP | FastAPI 쪽 MCP tool server |
| Infra | Docker Compose, AWS |

### 이 스택을 선택한 이유

- React는 과제 요구사항의 프론트엔드 조건을 만족합니다.
- NestJS는 게시판, 인증, 댓글, 태그, 검색 같은 일반 백엔드 기능을 구조적으로 만들기 좋습니다.
- FastAPI는 Python 기반 AI 기능, embedding, RAG, MCP, Agent 구현에 적합합니다.
- PostgreSQL은 일반 게시판 데이터와 운동 기록 데이터를 저장하기 좋고, pgvector를 붙여 의미 기반 검색까지 처리할 수 있습니다.
- JWT는 개인별 운동 기록과 AI 분석 히스토리를 구분하는 데 필요합니다.

### 서버 역할 원칙

- React는 NestJS만 호출합니다.
- NestJS는 사용자 인증과 게시판 도메인의 주인이 됩니다.
- FastAPI는 AI 기능 전용 내부 서버로 둡니다.
- FastAPI는 사용자가 직접 호출하지 않습니다.
- NestJS가 인증을 끝낸 뒤 FastAPI에 내부 요청을 보냅니다.

| 과제 요구사항 | 이 서비스에서의 구현 |
| --- | --- |

## 4. 사용자 시나리오

## 4.1 핵심 시나리오

1. 사용자가 회원가입한다.
2. 사용자가 로그인한다.
3. 사용자가 “오늘 가슴 운동” 게시글을 작성한다.
4. 운동 기록 표에 벤치프레스 60kg 8/8/7을 입력한다.
5. 게시글을 저장한다.
6. 게시글 상세 화면에서 AI 분석 버튼을 누른다.
7. AI가 이전 벤치프레스 기록을 찾는다.
8. AI가 지난 기록과 현재 기록을 비교한다.
9. AI가 “다음에는 60kg 8/8/8을 목표로 하라”고 추천한다.
10. 다른 사용자가 댓글로 응원한다.
11. 사용자는 내 기록 목록에서 성장 흐름을 확인한다.

## 4.2 데모용 예시 데이터

발표 데모를 위해 아래 데이터를 미리 넣어둡니다.

### 지난 기록

```text
제목: 지난 가슴 운동
운동: 벤치프레스
무게: 60kg
세트: 8회 / 7회 / 6회
메모: 마지막 세트가 많이 힘들었다.
```

### 오늘 기록

```text
제목: 오늘 가슴 운동
운동: 벤치프레스
무게: 60kg
세트: 8회 / 8회 / 7회
메모: 마지막 세트가 힘들었지만 지난번보다 나아진 것 같다.
```

### AI 기대 응답

```text
지난번 벤치프레스는 60kg으로 8/7/6이었고,
이번 기록은 60kg으로 8/8/7입니다.

총 반복 수가 21회에서 23회로 늘었기 때문에 발전하고 있습니다.
다만 아직 8/8/8을 모두 채우지는 못했으므로,
다음 운동에서는 무게를 올리기보다 60kg으로 8/8/8을 목표로 해보세요.

이 목표를 달성하면 그다음 운동에서 62.5kg에 도전해볼 수 있습니다.
```

## 5.4 운동 기록 작성 화면

### 목적

사용자가 오늘 운동 기록을 구조화해서 입력합니다.

### 입력 정보

- 제목
- 운동 날짜
- 운동 부위
- 메모
- 태그
- 운동 기록 표

### 운동 기록 표 컬럼

| 컬럼 | 설명 | 필수 |
| --- | --- | --- |
| 운동명 | 벤치프레스, 스쿼트 등 | 필수 |
| 무게 | kg 기준 | 필수 |
| 목표 반복 수 | 예: 8 | 선택 |
| 1세트 | 반복 수 | 필수 |
| 2세트 | 반복 수 | 선택 |
| 3세트 | 반복 수 | 선택 |
| 4세트 이상 | 추가 가능 | 선택 |
| 메모 | 자세, 난이도, 느낌 | 선택 |

### 동작

- 운동 행 추가
- 운동 행 삭제
- 세트 추가
- 세트 삭제
- 저장
- 임시 저장은 MVP에서 제외 가능

### 검증

- 운동명은 비어 있으면 안 됨
- 무게는 0보다 커야 함
- 반복 수는 0 이상이어야 함
- 최소 1개 운동은 있어야 함

## 8. 점진적 과부하 분석 규칙

AI가 아무 말이나 하지 않도록 기본 규칙을 정합니다.

## 8.1 기본 계산

### 총 반복 수

```text
totalReps = set1 + set2 + set3 + ...
```

예:

```text
8 + 8 + 7 = 23회
```

### 총 볼륨

```text
volume = weightKg * totalReps
```

예:

```text
60kg * 23회 = 1380kg
```

## 8.2 비교 기준

같은 운동명 또는 정규화된 운동명을 기준으로 최근 기록을 찾습니다.

비교 우선순위:

1. 같은 사용자
2. 같은 운동명
3. 같은 무게
4. 가장 최근 기록
5. 필요하면 최근 3개 기록

## 8.3 추천 규칙

### 규칙 1. 목표 반복 수를 모두 채운 경우

예:

```text
목표: 8/8/8
오늘: 8/8/8
```

추천:

```text
다음 운동에서 소폭 증량을 고려할 수 있습니다.
벤치프레스라면 2.5kg 정도 올려보는 것을 추천합니다.
```

### 규칙 2. 반복 수가 늘었지만 목표를 못 채운 경우

예:

```text
지난 기록: 60kg 8/7/6
오늘 기록: 60kg 8/8/7
목표: 8/8/8
```

추천:

```text
발전하고 있지만 아직 목표 반복 수를 모두 채우지는 못했습니다.
다음 운동에서는 같은 무게로 8/8/8을 목표로 해보세요.
```

### 규칙 3. 기록이 떨어진 경우

예:

```text
지난 기록: 60kg 8/8/7
오늘 기록: 60kg 7/6/5
```

추천:

```text
오늘은 이전보다 반복 수가 줄었습니다.
컨디션, 휴식, 자세를 점검하고 다음 운동에서는 같은 무게를 유지하는 것을 추천합니다.
```

### 규칙 4. 통증이 언급된 경우

예:

```text
메모: 어깨가 아팠다.
```

추천:

```text
통증이 있다면 무게를 올리기보다 자세와 휴식을 먼저 점검해보세요.
이 서비스는 의학적 진단을 제공하지 않으므로, 통증이 지속되면 전문가의 도움을 받는 것이 좋습니다.
```

### 규칙 5. 처음 기록인 경우

추천:

```text
이 운동의 첫 기록입니다.
다음 운동에서 같은 무게와 목표 반복 수를 다시 시도해 기준 기록을 만들어보세요.
```

## 9. RAG 기능 명세

## 9.1 RAG가 필요한 이유

이 서비스에서 RAG는 일반 지식 검색보다 “사용자의 이전 운동 기록 검색”에 더 중요합니다.

AI가 좋은 추천을 하려면 다음 정보가 필요합니다.

- 같은 사용자의 이전 기록
- 같은 운동의 최근 기록
- 지난 AI 분석 결과
- 사용자가 남긴 메모
- 유사한 운동 패턴

## 9.2 검색 대상

RAG 검색 대상:

- 운동 기록 게시글
- 운동별 세트 기록 요약
- 운동 메모
- 과거 AI 분석 결과

## 9.3 검색 방식

RAG는 두 가지 검색을 함께 사용합니다.

### 구조화 검색

DB에서 정확히 같은 운동명을 찾습니다.

예:

```text
사용자 A의 벤치프레스 최근 3개 기록
```

### 벡터 검색

pgvector에서 의미가 비슷한 기록을 찾습니다.

예:

```text
"마지막 세트가 힘들었다"
"무게를 올려도 될지 모르겠다"
"8/8/7에서 정체 중"
```

## 9.4 RAG 결과 예시

```json
{
  "currentRecord": {
    "exercise": "벤치프레스",
    "weightKg": 60,
    "sets": [8, 8, 7]
  },
  "retrievedRecords": [
    {
      "postId": 12,
      "date": "2026-06-02",
      "exercise": "벤치프레스",
      "weightKg": 60,
      "sets": [8, 7, 6],
      "memo": "마지막 세트가 힘들었다."
    }
  ]
}
```

## 9.5 RAG 성공 기준

- AI 분석에 이전 기록이 명시된다.
- 근거가 되는 게시글 ID 또는 날짜가 표시된다.
- 이전 기록이 없을 때도 자연스럽게 답변한다.
- 같은 사용자의 기록을 우선 사용한다.
- 다른 사용자의 기록을 사용할 경우 “참고용”임을 명확히 한다.

## 10. MCP 기능 명세

## 10.1 MCP의 역할

MCP는 AI가 외부 시스템을 호출할 수 있게 해주는 도구 연결 방식입니다.

이 서비스에서 MCP는 운동 기록 분석의 핵심 판단을 대신하지 않습니다. 대신 사용자가 입력한 운동명을 보강하고, 운동 부위나 장비 정보를 가져오는 보조 도구로 사용합니다.

## 10.2 선택할 외부 서비스

MVP에서는 외부 운동 정보 API 하나를 MCP tool로 연결합니다.

추천:

- wger Exercise API

사용 목적:

- 사용자가 입력한 운동명 정규화
- 운동 부위 정보 조회
- 장비 정보 조회
- 운동 카테고리 참고

예:

```text
사용자 입력: 벤치
정규화 결과: Bench Press
운동 부위: Chest
장비: Barbell, Bench
```

## 10.3 MCP Tool 목록

### Tool 1. `search_exercise_reference`

운동명을 외부 운동 데이터베이스에서 검색합니다.

입력:

```json
{
  "query": "bench press"
}
```

출력:

```json
{
  "name": "Bench Press",
  "category": "Chest",
  "equipment": ["Barbell", "Bench"],
  "muscles": ["Pectoralis major", "Triceps", "Anterior deltoid"]
}
```

### Tool 2. `normalize_exercise_name`

사용자가 입력한 운동명을 서비스 내부 표준명으로 바꿉니다.

입력:

```json
{
  "rawName": "벤치"
}
```

출력:

```json
{
  "normalizedName": "벤치프레스",
  "englishName": "Bench Press",
  "confidence": 0.86
}
```

## 10.4 MCP 성공 기준

- MCP tool이 실제 외부 API를 호출한다.
- 외부 API 실패 시 fallback 응답을 제공한다.
- tool call log가 저장된다.
- API key가 필요한 경우 환경 변수로 관리한다.
- AI 응답에 MCP 결과가 자연스럽게 반영된다.

## 10.5 MCP 실패 시 처리

외부 API가 실패해도 AI 분석 전체가 실패하지 않게 합니다.

예:

```text
외부 운동 정보 조회에는 실패했지만,
사용자의 이전 기록을 기준으로 분석을 진행했습니다.
```

## 11. Agent 기능 명세

## 11.1 Agent 이름

Workout Analysis Agent

## 11.2 Agent 목표

사용자의 운동 기록을 분석해 이전 기록과 비교하고, 다음 운동 목표를 추천합니다.

## 11.3 Agent가 사용할 도구

| 도구 | 역할 |
| --- | --- |
| `get_current_post` | 현재 게시글과 운동 기록 조회 |
| `get_user_previous_records` | 같은 사용자의 이전 운동 기록 조회 |
| `search_similar_records` | pgvector 기반 유사 기록 검색 |
| `search_exercise_reference` | MCP로 외부 운동 정보 조회 |
| `generate_recommendation` | LLM으로 최종 분석 생성 |
| `save_analysis_result` | 분석 결과 저장 |

## 11.4 Agent 실행 단계

1. 현재 게시글을 가져온다.
2. 운동 기록 표를 파싱한다.
3. 각 운동명을 정규화한다.
4. 같은 사용자의 이전 운동 기록을 조회한다.
5. 필요하면 RAG로 유사 기록을 검색한다.
6. 필요하면 MCP tool로 운동 정보를 조회한다.
7. 점진적 과부하 규칙에 따라 초안 분석을 만든다.
8. LLM이 사용자 친화적 문장으로 다듬는다.
9. 안전 문구를 검증한다.
10. 분석 결과를 저장한다.
11. NestJS에 결과를 반환한다.

## 11.5 Agent 제한 조건

- 최대 단계 수: 8단계
- 최대 MCP tool 호출: 2회
- 최대 RAG 검색: 2회
- 최대 실행 시간: 30초
- 게시글 자동 발행 금지
- 의학적 진단 금지
- 통증 언급 시 주의 문구 포함

## 15. 비기능 요구사항

## 15.1 보안

- 비밀번호는 해시로 저장한다.
- JWT secret은 환경 변수로 관리한다.
- OpenAI API key는 서버 환경 변수로만 관리한다.
- FastAPI는 외부에서 직접 호출하지 않도록 한다.
- AI 요청에는 rate limit을 둔다.
- 게시글/댓글 입력값은 XSS를 고려해 렌더링한다.

## 15.2 성능

- 게시글 목록은 페이징한다.
- 운동명, 작성자, 날짜에는 인덱스를 고려한다.
- AI 분석은 중복 실행을 막는다.
- 분석 결과가 있으면 재사용한다.

## 15.3 안정성

- AI 분석 실패 시 게시글 저장은 유지한다.
- MCP 실패 시 RAG와 기존 기록만으로 분석한다.
- OpenAI timeout을 설정한다.
- Agent 실행 로그를 남긴다.

## 15.4 사용성

- 운동 기록 입력은 표 형태로 제공한다.
- AI 결과는 전문 용어보다 쉬운 말로 보여준다.
- 무게 증량 여부를 명확히 보여준다.
- 통증 관련 문구는 조심스럽게 표현한다.

## 20. 한계와 주의사항

## 20.1 의학적 조언 금지

AI는 부상 진단을 하지 않습니다.

표현 원칙:

- “통증이 있다면 무게를 올리지 마세요.”
- “전문가와 상담해보세요.”
- “이 서비스는 의학적 진단을 제공하지 않습니다.”

## 20.2 운동명 정규화의 한계

사용자는 같은 운동을 여러 방식으로 입력할 수 있습니다.

예:

- 벤치
- 벤치프레스
- Barbell Bench Press
- 가슴 벤치

MVP에서는 자주 쓰는 운동명부터 정규화합니다.

## 20.3 AI 판단의 한계

AI는 사용자의 자세, 컨디션, 수면, 식사, 통증 정도를 정확히 알 수 없습니다.

따라서 추천은 절대적인 처방이 아니라 참고 목표로 표현해야 합니다.

## 22. 최종 정리

이 프로젝트의 핵심은 운동 기록을 단순히 저장하는 것이 아니라, 기록을 해석해서 다음 행동으로 연결하는 것입니다.

MVP에서 가장 중요한 흐름은 다음입니다.

```text
운동 기록 작성
-> 이전 기록 검색
-> 발전 여부 비교
-> 다음 목표 추천
-> 게시판에서 공유와 응원
```

기술적으로는 일반 게시판 기능을 NestJS가 안정적으로 담당하고, FastAPI가 AI 분석을 전담하는 구조로 갑니다.

과제 평가에서 가장 잘 보여줘야 할 것은 다음 세 가지입니다.

1. 게시판 기본 기능이 완성되어 있다.
2. AI가 이전 운동 기록을 근거로 분석한다.
3. RAG, MCP, Agent가 서비스 주제 안에서 자연스럽게 연결되어 있다.
