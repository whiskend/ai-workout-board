# 2026년 6월 16일 화요일 학습 문서

> 주제: 현재 AI 운동 기록 게시판을 설계자 관점에서 읽기  
> 목표: 코드를 더 치기 전에, 이 프로젝트가 어떤 구조로 나뉘어 있고 앞으로 기능을 추가할 때 무엇을 결정해야 하는지 이해한다.

---

# 0. 오늘 문서의 목적

오늘은 새 문법을 외우는 날이 아니다.

오늘은 설계자로서 프로젝트를 보는 연습을 한다.

여기서 말하는 설계자는 이런 사람이다.

```text
어떤 기능을 만들 때
어느 서버가 맡을지,
어느 API가 필요할지,
DB에 저장할지,
화면에만 보여줄지,
권한 검사는 어디서 할지,
기존 구조를 깨뜨리지 않는지 판단하는 사람
```

즉 설계자는 코드를 한 줄씩 치기 전에 먼저 아래 질문을 던진다.

```text
이 기능은 어디에 들어가야 하지?
이 데이터는 누가 가지고 있어야 하지?
이 요청은 로그인해야 가능한가?
이 결과는 저장해야 하나?
기존 API 응답을 바꿔도 되나?
지금 작업 범위가 너무 커진 건 아닌가?
```

이 문서는 현재 프로젝트를 기준으로 그 질문들을 공부하기 위한 문서다.

---

# 1. 현재 프로젝트를 한 문장으로 다시 보기

현재 프로젝트는 이것이다.

```text
운동 기록을 게시글처럼 남기고,
이전 운동 기록과 비교해서
AI가 다음 운동 목표를 추천해주는 게시판
```

핵심은 단순 게시판이 아니다.

핵심은 아래 흐름이다.

```text
오늘 운동 기록 작성
        ↓
이전 기록 검색
        ↓
현재 기록과 비교
        ↓
다음 운동 목표 추천
```

그래서 설계자로서 제일 먼저 봐야 하는 것은 기능 목록이 아니라 **데이터 흐름**이다.

---

# 2. 현재 전체 구조

현재 프로젝트는 크게 4개 영역으로 나뉜다.

```text
React Frontend
-> NestJS Backend
-> PostgreSQL Database
-> FastAPI AI Server
```

조금 더 풀면 이렇다.

```text
React
- 사용자가 보는 화면
- 입력 폼
- 버튼
- API 요청 보내기
- 결과 화면에 표시하기

NestJS
- HTTP API 제공
- 로그인/JWT 검사
- 작성자 권한 검사
- PostgreSQL 조회/저장
- FastAPI 호출 조율

PostgreSQL
- 사용자
- 게시글
- 운동 종목
- 세트 기록
- 댓글
- 태그

FastAPI
- AI 분석
- 운동명 정규화 tool
- Agent workflow
- OpenAI 호출 또는 rule-based fallback
```

현재 중요한 원칙은 이것이다.

```text
React는 FastAPI를 직접 호출하지 않는다.
AI 분석 요청도 반드시 NestJS를 거친다.
```

이 결정은 설계적으로 중요하다.

왜냐하면 React가 FastAPI를 직접 호출하면 인증, 권한, DB 조회 책임이 흩어진다.

현재 프로젝트에서는 NestJS가 사용자를 확인하고, DB에서 현재 기록과 이전 기록을 찾고, FastAPI에는 분석에 필요한 데이터만 넘긴다.

---

# 3. 설계 포인트 1: 책임 분리

설계자가 제일 먼저 공부해야 하는 것은 책임 분리다.

책임 분리는 쉽게 말하면:

```text
누가 뭘 해야 하는지 정하는 것
```

현재 프로젝트에서 책임은 이렇게 나뉜다.

| 영역 | 책임 | 대표 파일 |
| --- | --- | --- |
| React Page | 화면과 사용자 입력 | `frontend/src/pages/*.tsx` |
| React API | 백엔드 요청 함수 | `frontend/src/api/*.ts` |
| NestJS Controller | HTTP 요청 받기 | `backend/src/posts/posts.controller.ts` |
| NestJS Service | 실제 작업 처리 | `backend/src/posts/posts.service.ts` |
| Prisma Schema | DB 구조 정의 | `backend/prisma/schema.prisma` |
| FastAPI Router | AI 서버 API 주소 | `ai-server/app/routers/*.py` |
| FastAPI Service | AI 분석 로직 | `ai-server/app/services/*.py` |

예를 들어 게시글 상세 화면에서 `AI 분석` 버튼을 누르면:

```text
PostDetailPage.tsx
-> analyzePost(postId)
-> POST /posts/:id/analyze
-> PostsController.analyze()
-> PostsService.analyze()
-> AiService.analyzePost()
-> FastAPI
-> Agent workflow
-> React 화면에 결과 표시
```

이 흐름에서 설계자가 확인해야 할 것은:

```text
화면이 DB를 직접 만지고 있지 않은가?
Controller가 너무 많은 일을 하고 있지 않은가?
Service가 권한 검사 없이 DB를 수정하고 있지 않은가?
FastAPI가 사용자 인증 책임까지 가져가고 있지 않은가?
```

현재 구조에서는 이 책임이 꽤 잘 나뉘어 있다.

---

# 4. 설계 포인트 2: API는 약속이다

API는 그냥 함수가 아니다.

API는 프론트엔드와 백엔드 사이의 약속이다.

예를 들어:

```text
POST /posts/:id/analyze
```

이 API는 이런 약속이다.

```text
프론트엔드:
게시글 id를 URL에 넣어서 요청할게.
Authorization header에 accessToken도 넣을게.

백엔드:
토큰을 확인하고,
그 사용자가 이 글 작성자인지 확인하고,
이전 운동 기록을 찾아서,
AI 분석 결과를 JSON으로 돌려줄게.
```

그래서 설계자는 API를 볼 때 아래를 확인해야 한다.

```text
1. URL은 무엇인가?
2. HTTP method는 무엇인가?
3. 로그인이 필요한가?
4. URL param은 무엇인가?
5. query string은 무엇인가?
6. body에는 무엇이 들어가는가?
7. 응답으로 무엇을 주는가?
8. 실패하면 어떤 에러가 나가는가?
```

현재 게시글 API를 보면:

```text
GET    /posts
GET    /posts/:id
POST   /posts
PATCH  /posts/:id
DELETE /posts/:id
POST   /posts/:id/analyze
POST   /posts/:id/comments
DELETE /posts/:postId/comments/:commentId
```

여기서 설계자로 볼 포인트는:

```text
조회는 공개 API인가?
작성/수정/삭제/분석/댓글은 로그인 API인가?
수정/삭제/분석은 작성자 본인만 가능한가?
댓글 삭제는 댓글 작성자만 가능한가?
```

즉 API 설계는 단순히 주소를 정하는 일이 아니다.

```text
누가 요청할 수 있는지,
무슨 데이터를 줄지,
어디까지 책임질지를 정하는 일
```

이다.

# 5. 설계 포인트 3: DB는 서비스를 어떻게 기억할지 정하는 곳이다

현재 DB 구조는 운동 기록 게시판의 핵심 생각을 그대로 담고 있다.

```text
User
  └── Post
        ├── Exercise
        │     └── ExerciseSet
        ├── Comment
        └── PostTag
              └── Tag
```

이 구조를 사람 말로 바꾸면:

```text
사용자는 여러 게시글을 쓸 수 있다.
게시글 하나에는 여러 운동 종목이 들어갈 수 있다.
운동 종목 하나에는 여러 세트가 들어갈 수 있다.
게시글에는 댓글이 달릴 수 있다.
게시글에는 여러 태그가 붙을 수 있다.
태그 하나는 여러 게시글에 붙을 수 있다.
```

설계자가 DB를 볼 때 중요한 질문은 이것이다.

```text
이 데이터는 정말 저장해야 하나?
하나의 데이터가 여러 개를 가질 수 있나?
삭제되면 같이 삭제되어야 하나?
검색해야 하는 값인가?
중복을 허용해도 되나?
```

## 5.1 Post와 Exercise를 나눈 이유

게시글 하나에 운동 기록을 그냥 긴 문자열로 저장할 수도 있었다.

예:

```text
memo = "벤치프레스 60kg 8회 8회 7회"
```

하지만 이렇게 하면 AI 분석과 검색이 어렵다.

그래서 현재는 나눠서 저장한다.

```text
Post
- 제목
- 날짜
- 운동 부위
- 메모

Exercise
- 운동명
- 정규화된 운동명
- 무게
- 목표 반복 수

ExerciseSet
- 몇 세트인지
- 몇 회 했는지
- 체감 난이도
```

이렇게 나누면 좋은 점:

```text
운동명으로 검색 가능
이전 같은 운동 기록 비교 가능
세트별 반복 수 비교 가능
나중에 그래프 만들기 쉬움
```

## 5.2 normalizedName을 둔 이유

사람은 같은 운동을 다르게 쓴다.

```text
벤치
벤치프레스
bench press
bench
```

사람 눈에는 비슷하지만 DB는 다르게 본다.

그래서 `normalizedName`을 둔다.

```text
벤치 -> 벤치프레스
bench press -> 벤치프레스
벤치프레스 -> 벤치프레스
```

설계적으로 이 필드는 RAG 검색 품질과 연결된다.

이전 기록을 찾을 때:

```text
exerciseName이 정확히 같은 글
```

만 찾는 것보다:

```text
normalizedName이 같은 글
```

을 찾는 것이 더 안정적이다.

## 5.3 Tag와 PostTag를 나눈 이유

태그는 게시글과 다대다 관계다.

```text
게시글 하나에 태그 여러 개
태그 하나가 여러 게시글에 사용됨
```

그래서 중간 테이블인 `PostTag`가 필요하다.

```text
Post
Tag
PostTag
```

이 구조는 처음엔 복잡해 보이지만, 나중에 아래 기능을 만들기 좋다.

```text
#가슴 태그가 붙은 글만 보기
#벤치프레스 태그 인기순 보기
태그별 운동 기록 통계
```

# 6. 설계 포인트 4: 인증과 권한은 다르다

초보자가 많이 헷갈리는 부분이다.

```text
인증
= 너 누구야?

권한
= 네가 이걸 해도 돼?
```

현재 프로젝트에서 인증은 JWT Guard가 한다.

```text
@UseGuards(JwtAuthGuard)
```

이게 붙은 API는 먼저 로그인 토큰을 검사한다.

예:

```text
POST /posts
PATCH /posts/:id
DELETE /posts/:id
POST /posts/:id/analyze
POST /posts/:id/comments
DELETE /posts/:postId/comments/:commentId
```

하지만 인증만으로는 부족하다.

로그인한 사용자라도 남의 게시글을 수정하면 안 된다.

그래서 Service에서 권한 검사를 한다.

```text
post.authorId !== userId
```

이 말은:

```text
DB에 저장된 게시글 작성자 id와
요청을 보낸 로그인 사용자 id가 다르면
막는다.
```

설계자로서 봐야 할 핵심:

```text
Controller에서 토큰을 검사하는가?
Service에서 실제 소유자 권한을 검사하는가?
단순히 로그인만 했다고 모든 수정/삭제를 허용하지는 않는가?
```

인증과 권한을 나눠서 생각해야 사고가 줄어든다.

# 7. 설계 포인트 5: RAG, MCP, Agent를 억지로 섞지 않기

이 프로젝트에서 AI 기능은 세 단어가 같이 나온다.

```text
RAG
MCP
Agent workflow
```

이걸 한 덩어리로 보면 헷갈린다.

설계자는 각각의 책임을 분리해서 봐야 한다.

## 7.1 RAG

RAG는 쉽게 말하면:

```text
AI에게 답변시키기 전에
관련 있는 기존 데이터를 먼저 찾아서
그 데이터를 같이 넘기는 구조
```

현재 프로젝트에서는:

```text
현재 운동 기록
        ↓
같은 사용자 + 같은 운동명 + 과거 날짜 + 최근 3개 기록 검색
        ↓
FastAPI 분석 요청에 previousPosts로 포함
```

여기까지가 RAG의 핵심이다.

즉 현재 프로젝트에서 RAG의 중요한 부분은:

```text
이전 운동 기록을 찾는 검색 조건
```

이다.

## 7.2 MCP-style tool

MCP는 원래 LLM이 외부 도구를 호출할 수 있게 하는 프로토콜이다.

현재 프로젝트는 완전한 독립 MCP 서버까지는 가지 않았다.

대신 FastAPI 내부에 tool 역할을 하는 구조를 만들었다.

현재 tool:

```text
normalize_exercise_name
```

역할:

```text
벤치, bench, bench press 같은 표현을
벤치프레스로 맞춘다.
```

설계적으로 이 tool은 RAG 검색 품질을 높인다.

왜냐하면 이전 기록을 찾을 때 운동명이 조금 달라도 같은 운동으로 묶을 수 있기 때문이다.

## 7.3 Agent workflow

Agent workflow는 완전 자율 AI가 아니다.

현재 프로젝트에서는 정해진 순서로 움직이는 분석 흐름이다.

```text
1. 현재 기록 입력 받기
2. 운동명 정규화 tool 결과 받기
3. 이전 기록 검색 결과 받기
4. AI 분석 실행
5. 결과 반환
```

이 구조의 설계 포인트:

```text
AI가 멋대로 움직이는 게 아니라
우리가 정한 순서대로 tool과 분석을 실행한다.
```

발표할 때는 이렇게 말하면 된다.

```text
완전 자율 Agent까지 구현한 것은 아니고,
운동 기록 분석에 필요한 순서를 workflow로 고정했습니다.
현재 기록 입력, tool 호출, 이전 기록 검색, 분석, 결과 반환 단계를 명시적으로 기록해서
분석 과정이 화면에 드러나도록 했습니다.
```

# 8. 설계 포인트 6: 저장할 것과 저장하지 않을 것

설계에서 매우 중요한 질문이다.

```text
이 결과를 DB에 저장할까?
그냥 화면에만 보여줄까?
```

현재 프로젝트의 결정:

```text
AI 분석 결과는 DB에 저장하지 않는다.
화면에만 보여준다.
```

왜 이렇게 했을까?

저장하려면 새 테이블이 필요하다.

예:

```text
AnalysisResult
- id
- postId
- summary
- recommendation
- nextGoal
- createdAt
```

그러면 해야 할 일이 늘어난다.

```text
DB schema 변경
migration
저장 API
조회 API
화면에서 과거 분석 결과 표시
분석 결과가 최신 기록과 안 맞을 때 처리
```

현재 목표는 발표 가능한 AI 분석 흐름을 만드는 것이다.

그래서 지금은:

```text
게시글/운동기록/댓글/태그는 저장
AI 분석 결과는 저장하지 않고 응답으로만 표시
```

이 결정은 좋은 설계 공부 포인트다.

왜냐하면 모든 데이터를 저장하는 것이 항상 좋은 설계는 아니기 때문이다.

# 9. 설계 포인트 7: fallback은 데모 안정성을 위한 설계다

현재 FastAPI는 OpenAI 호출을 시도한다.

하지만 OpenAI API 키가 없거나, 네트워크가 안 되거나, 호출이 실패할 수 있다.

그때 그냥 에러를 내면 데모가 끊긴다.

그래서 현재 구조는:

```text
OpenAI Responses API 호출 시도
        ↓
성공하면 OpenAI 기반 분석
        ↓
실패하면 rule-based 분석
```

이것을 fallback이라고 한다.

설계자로서 볼 포인트:

```text
외부 서비스는 실패할 수 있다고 가정해야 한다.
실패해도 핵심 흐름이 완전히 죽지 않게 해야 한다.
```

이 프로젝트에서 핵심 흐름은:

```text
운동 기록을 분석하고 다음 목표를 보여주는 것
```

OpenAI는 분석 품질을 높이는 수단이다.

하지만 발표나 테스트에서는 최소한 rule-based로라도 결과가 나와야 한다.

그래서 fallback은 단순 예외처리가 아니라 설계 결정이다.

# 10. 설계 포인트 8: 프론트엔드는 화면만 예쁘게 만드는 곳이 아니다

프론트엔드도 설계가 있다.

현재 프론트는 대략 이렇게 나뉜다.

```text
frontend/src/pages
- 실제 페이지
- 라우팅 단위
- 화면 흐름 담당

frontend/src/api
- 백엔드 API 호출 함수
- fetch, token, Authorization header 처리

frontend/src/types
- API 응답 데이터 타입
- 화면에서 사용할 데이터 구조

frontend/src/components
- 재사용 UI
- Layout, Sidebar, PageHeader, Button, Card 등
```

설계자로서 프론트에서 볼 질문:

```text
이 코드는 페이지에 있어야 하나, 공통 컴포넌트로 빼야 하나?
API 호출 코드가 화면 코드에 너무 섞여 있지 않은가?

accessToken 처리가 여러 곳에 흩어져 있지 않은가?
모바일 화면에서 구조가 깨지지 않는가?
```

이번 UI 개선에서 중요한 결정은:

```text
페이지 전체 구조는 AppLayout이 담당
왼쪽/상단 네비게이션은 Sidebar가 담당
각 화면 제목은 PageHeader가 담당
반복 UI는 Button/Card/Badge/Input 같은 컴포넌트가 담당
```

이렇게 나누면 나중에 화면을 고칠 때 한 곳만 고치면 된다.

# 14. 예시: “AI 분석 결과 저장 기능”을 설계한다면

```text
사용자가 AI 분석 결과를 나중에 다시 보고 싶다.
```

설계자가 먼저 생각할 것:

```text
이 결과를 DB에 저장해야 하나?
분석할 때마다 새로 저장할까?

여러 번 분석한 기록을 모두 저장할까?

작성자만 볼 수 있게 할까?
```

가능한 설계안:


```text

```

가 안전하다.

```text
B안
```

이 적당하다.

이 예시에서 배울 점:

```text
기능 하나를 추가한다는 것은 버튼 하나를 추가하는 게 아니다.

```

# 17. 설계자로 발표할 때 쓸 수 있는 설명

아래는 발표나 팀 공유 때 그대로 말해도 되는 버전이다.

```text
우리 서비스는 React, NestJS, PostgreSQL, FastAPI로 나누었습니다.

React는 사용자가 운동 기록을 작성하고 AI 분석 결과를 보는 화면을 담당합니다.
NestJS는 인증, 권한 검사, 게시글 CRUD, DB 조회를 담당합니다.
PostgreSQL은 사용자, 게시글, 운동 종목, 세트, 댓글, 태그를 저장합니다.
FastAPI는 AI 분석, 운동명 정규화 tool, Agent workflow를 담당합니다.

AI 분석 요청은 React가 FastAPI를 직접 호출하지 않고 NestJS를 거치게 했습니다.
그 이유는 JWT 인증과 작성자 권한 검사, 그리고 이전 운동 기록 조회를 NestJS에서 책임지기 위해서입니다.

RAG는 현재 사용자의 같은 운동명 기록 중 현재 글보다 과거 날짜의 최근 3개를 찾아 분석에 넣는 방식으로 최소 구현했습니다.
MCP는 독립 서버 대신 FastAPI 내부 tool endpoint로 운동명 정규화를 구현했습니다.
Agent는 완전 자율 에이전트가 아니라 현재 기록 입력, tool 호출, 이전 기록 검색 결과 수신, 분석 실행, 결과 반환의 5단계 workflow로 구현했습니다.

AI 분석 결과는 DB에 저장하지 않고 화면에만 표시합니다.
현재 목표는 분석 이력 관리보다 AI 분석 흐름을 안정적으로 데모하는 것이기 때문에 DB schema 변경을 피했습니다.
```

# 19. 오늘의 결론

설계자는 코드를 안 치는 사람이 아니다.

설계자는 코드를 치기 전에:

```text
책임을 나누고
데이터 흐름을 정하고
저장 여부를 판단하고
권한 위치를 확인하고
작업 범위를 자르는 사람
```

이다.

현재 프로젝트에서 설계자로 가장 먼저 익혀야 할 감각은 이것이다.

```text
화면에서 시작된 요청이
어느 서버를 거쳐
어떤 DB 데이터를 보고
어떤 AI 서버 작업을 거쳐
다시 화면으로 돌아오는지
끝까지 그릴 수 있어야 한다.
```

이걸 그릴 수 있으면, 다음 기능을 만들 때도 덜 흔들린다.
