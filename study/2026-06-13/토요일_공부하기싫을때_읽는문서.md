```text

이걸 며칠 사이에 한 번에 보고 있다.

```

```text
금요일에는 React가 NestJS한테 말을 걸게 만들었다.

```

딱 이거다.

# 1. 오늘의 진짜 목표

오늘 목표는 멋진 AI가 아니다.

오늘 목표는 이것이다.

```text
AI 분석 버튼을 눌렀을 때
뭐라도 분석 결과가 화면에 나온다.
```

정말로 이 정도면 된다.

예를 들면 이런 응답이면 충분하다.

```json
{
  "summary": "오늘 가슴 운동 기록입니다.",
  "recommendation": "다음 운동에서는 같은 무게로 목표 반복 수를 채워보세요.",
  "nextGoal": "벤치프레스 60kg 8/8/8을 목표로 해보세요.",
  "referencedPostCount": 1
}
```

이게 화면에 나오면 오늘은 성공이다.

GPT가 엄청 똑똑하게 말하지 않아도 된다.

일단 길이 뚫려야 한다.

```text
길이 뚫리면 나중에 GPT를 태우면 된다.
길이 안 뚫리면 GPT를 붙여도 어디서 터졌는지 모른다.
```

---

# 2. 오늘 구현을 게임처럼 보면

오늘은 보스전이 아니라 배관 연결이다.

```text
1번 파이프: React -> NestJS
2번 파이프: NestJS -> FastAPI
3번 파이프: FastAPI -> NestJS
4번 파이프: NestJS -> React
```

금요일에 1번 파이프는 뚫었다.

오늘은 2, 3, 4번 파이프를 뚫는 날이다.

완성되면 이런 일이 생긴다.

```text
브라우저 버튼 하나
        ↓
백엔드가 현재 게시글 찾음
        ↓
백엔드가 이전 운동 기록 찾음
        ↓
AI 서버에 보냄
        ↓
AI 서버가 분석 결과 만듦
        ↓
화면에 결과 표시
```

이건 꽤 멋있다.

왜냐하면 이제 단순 게시판이 아니라:

```text
내 기록을 읽고 반응하는 서비스
```

가 되기 시작하기 때문이다.

---

# 3. 오늘 딱 세 덩어리만 하면 된다

오늘 작업을 크게 보면 세 덩어리다.

```text
1. FastAPI에 분석 API 만들기
2. NestJS에서 그 API 호출하기
3. React 화면에 분석 버튼 붙이기
```

이걸 더 쉽게 말하면:

```text
FastAPI: 분석 결과 만들어주는 곳
NestJS: 게시글 찾고 AI 서버에 심부름 보내는 곳
React: 버튼 누르고 결과 보여주는 곳
```

각자 역할이 다르다.

```text
React는 DB를 모른다.
FastAPI도 로그인 사용자를 직접 모른다.
NestJS가 중간에서 다 조율한다.
```

그래서 구조가 이렇게 된다.

```text
React
  "이 글 분석해줘"
        ↓
NestJS
  "이 사용자가 쓴 글 맞나?"
  "이전 기록 있나?"
  "AI 서버야 이거 분석해줘"
        ↓
FastAPI
  "분석 결과 만들었어"
        ↓
NestJS
  "React야 결과 여기 있어"
        ↓
React
  화면에 표시
```

# 4. 오늘 외울 단어는 5개만

오늘 모든 걸 이해하려고 하지 말고 이 5개만 잡는다.

## 4.1 FastAPI

Python으로 만든 AI 전용 서버.

```text
GPT 호출
운동 기록 분석
RAG
MCP
Agent
```

이런 AI 쪽 기능이 들어갈 자리다.

## 4.2 Schema

FastAPI에서 요청/응답 데이터 모양을 정하는 것.

NestJS의 DTO랑 비슷하다.

```text
NestJS DTO
FastAPI Schema
```

둘 다 뜻은 비슷하다.

```text
데이터는 이 모양으로 들어와야 한다.
```

## 4.3 Router

FastAPI에서 API 주소를 모아두는 곳.

NestJS의 Controller랑 비슷하다.

```text
NestJS Controller
FastAPI Router
```

## 4.4 Service

실제 일을 하는 곳.

```text
분석 문장 만들기
FastAPI 호출하기
DB에서 이전 기록 찾기
```

이런 작업은 Service가 한다.

## 4.5 Analyze

분석한다는 뜻이다.

오늘 만들 핵심 API는 이거다.

```text
POST /posts/:id/analyze
```

뜻:

```text
이 게시글을 AI로 분석해줘.
```

  "service": "ai-server"

가 나오면 첫 번째 성공이다.

이건 작아 보이지만 중요하다.

왜냐하면:

```text
AI 서버가 살아 있다.
```

는 뜻이기 때문이다.

서버가 살아 있어야 분석도 한다.

# 6. 두 번째 성공

두 번째 성공은 FastAPI가 혼자 분석 결과를 만드는 것이다.

```bash
curl -X POST http://localhost:8000/analysis/demo \
  -H "Content-Type: application/json" \
  -d '{...}'
```

여기서 응답이 이렇게 나오면 된다.

```json
{
  "summary": "...",
  "recommendation": "...",
  "nextGoal": "...",
  "referencedPostCount": 0
}
```

이 단계에서는 NestJS도 React도 신경 쓰지 않는다.

오직 이것만 본다.

```text
FastAPI가 요청을 받고 응답을 준다.
```

---

# 7. 세 번째 성공

세 번째 성공은 NestJS가 FastAPI를 호출하는 것이다.

```bash
curl -X POST http://localhost:3000/posts/1/analyze \
  -H "Authorization: Bearer accessToken"
```

이게 성공하면 정말 큰 단계다.

왜냐하면 이 흐름이 된다는 뜻이기 때문이다.

```text
curl
-> NestJS
-> DB 조회
-> FastAPI 호출
-> 분석 결과 반환
```

금요일까지는 NestJS가 DB만 봤다.

토요일부터는 NestJS가 다른 서버도 부른다.

이 순간부터 프로젝트가 조금 더 진짜 서비스처럼 보이기 시작한다.

# 9. 막힐 때 보는 표

## 9.1 FastAPI가 안 된다

확인:

```bash
curl http://localhost:8000/health
```

안 되면:

```text
AI 서버가 안 켜진 것이다.
```

## 9.2 NestJS 분석 API가 안 된다

확인:

```bash
curl -X POST http://localhost:3000/posts/1/analyze \
  -H "Authorization: Bearer accessToken"
```

안 되면 둘 중 하나다.

```text
token 문제
또는 NestJS -> FastAPI 호출 문제
```

## 9.3 브라우저만 안 된다

확인:

```text
프론트 API 함수 이름이 맞는가?
token이 localStorage에 있는가?
버튼 onClick이 handleAnalyze에 연결됐는가?
```

브라우저만 안 되면 FastAPI 문제가 아닐 가능성이 높다.
