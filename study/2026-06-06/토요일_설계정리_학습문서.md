# 토요일 설계 정리 학습 문서

> 날짜: 2026년 6월 6일 토요일  
> 목적: 본격적인 구현 전에 MVP 범위, ERD, API, 폴더 구조, GitHub 이슈, 브랜치 전략, Docker Compose 방향을 확정한다.  
> 이 문서는 이미 토요일에 진행한 내용을 다시 공부하기 위한 복습용 문서입니다.

---

# 1. 토요일의 핵심 목표

토요일의 목표는 코드를 많이 치는 것이 아니었습니다.

토요일의 진짜 목표는 이것이었습니다.

```text
월요일부터 "뭘 만들지?"를 고민하지 않고,
"정해진 순서대로 만들기"만 할 수 있게 준비하는 것
```

즉, 토요일은 구현의 날이라기보다 **결정의 날**입니다.

개발을 처음 시작하면 바로 코드를 치고 싶어집니다. 하지만 아무것도 정하지 않은 상태에서 코드를 치기 시작하면 곧 이런 문제가 생깁니다.

- 어떤 기능부터 만들어야 하는지 모른다.
- DB 테이블을 어떻게 만들어야 하는지 모른다.
- API 주소를 어떻게 정해야 하는지 모른다.
- 프론트엔드와 백엔드가 어떤 데이터를 주고받아야 하는지 모른다.
- GitHub Issue를 어떻게 나눠야 할지 모른다.
- 브랜치를 어떻게 나눠야 할지 모른다.
- Docker를 어디까지 써야 할지 모른다.

그래서 토요일에는 기능 구현 전에 필요한 기본 결정을 먼저 닫았습니다.

토요일에 정한 결론은 아래와 같습니다.

```text
서비스 주제:
운동 기록 인증 게시판

핵심 기능:
사용자가 운동 기록을 게시글로 남기고,
AI가 이전 운동 기록과 비교해 다음 운동 목표를 추천한다.

기술 스택:
React + NestJS + FastAPI + PostgreSQL + pgvector + Prisma + Docker Compose
```

# 2. 토요일 작업 전체 지도

토요일 작업은 크게 7가지였습니다.

```text
1. MVP 범위 확정
2. ERD 초안 확정
3. API 목록 확정
4. 프로젝트 폴더 구조 확정
5. GitHub 이슈 생성
6. 브랜치 전략 정하기
7. Docker Compose 방향 정하기
```

이 7가지는 각각 따로 떨어진 작업처럼 보이지만 사실 하나의 흐름입니다.

```text
MVP 범위
  ↓
필요한 데이터 결정
  ↓
ERD 작성
  ↓
필요한 기능 결정
  ↓
API 목록 작성
  ↓
폴더 구조 결정
  ↓
GitHub 이슈로 작업 쪼개기
  ↓
브랜치 전략으로 작업 흐름 정리
  ↓
Docker Compose로 개발 환경 준비
```

즉, 토요일 작업은 **아이디어를 실제 개발 작업으로 바꾸는 과정**이었습니다.

# 3. MVP 범위 확정

## 3.1 MVP가 뭔가?

MVP는 `Minimum Viable Product`의 줄임말입니다.

한국어로 풀면:

```text
최소 기능 제품
```

여기서 중요한 단어는 **최소**입니다.

MVP는 "내가 상상한 모든 기능을 다 넣은 완성형 서비스"가 아닙니다.

MVP는:

```text
발표에서 서비스의 핵심 가치를 보여줄 수 있는 가장 작은 버전
```

입니다.

우리 서비스의 핵심 가치는 단순 게시판이 아닙니다.

우리 서비스의 핵심 가치는 이것입니다.

```text
운동 기록을 저장하면,
AI가 이전 기록과 비교해서,
다음 운동 목표를 추천해준다.
```

따라서 MVP에 반드시 들어가야 하는 것은 아래 흐름입니다.

```text
회원가입/로그인
  ↓
운동 기록 게시글 작성
  ↓
운동 기록 표 저장
  ↓
게시글 조회
  ↓
이전 운동 기록 검색
  ↓
AI 분석
  ↓
다음 운동 목표 추천
```

## 3.2 이번 과제 MVP에 포함할 기능

토요일에 확정한 MVP 필수 기능은 아래입니다.

### 일반 게시판 기능

- 회원가입
- 로그인
- JWT 인증
- 운동 기록 게시글 작성
- 게시글 목록 조회
- 게시글 상세 조회
- 게시글 수정
- 게시글 삭제
- 댓글 작성
- 댓글 삭제
- 태그 등록
- 태그 조회
- 검색
- 페이징

### 운동 기록 특화 기능

- 운동 기록 표 입력
- 운동명 저장
- 무게 저장
- 세트별 반복 수 저장
- 운동별 이전 기록 조회
- 내 운동 기록 목록 조회

예를 들어 사용자가 아래처럼 입력할 수 있어야 합니다.

```text
제목: 오늘 가슴 운동
운동명: 벤치프레스
무게: 60kg
세트: 8회 / 8회 / 7회
메모: 마지막 세트가 힘들었다.
```

이걸 그냥 긴 문자열 하나로 저장하면 AI가 비교하기 어렵습니다.

그래서 운동 기록은 표 형태의 데이터로 나눠 저장해야 합니다.

```text
게시글
  └─ 운동 1: 벤치프레스
       ├─ 세트 1: 60kg 8회
       ├─ 세트 2: 60kg 8회
       └─ 세트 3: 60kg 7회
```

### AI 기능

- RAG 기반 이전 운동 기록 검색
- AI 분석 버튼
- 이전 기록과 현재 기록 비교
- 발전 여부 판단
- 다음 운동 목표 추천
- 운동 기록 요약
- MCP tool 최소 1개
- Agent workflow 최소 구현

여기서 중요한 점은 AI 기능도 막연한 챗봇이 아니라는 것입니다.

우리 과제의 AI 기능은 아래처럼 구체적이어야 합니다.

```text
현재 기록:
벤치프레스 60kg 8/8/7

이전 기록:
벤치프레스 60kg 8/7/6

AI 판단:
총 반복 수가 21회에서 23회로 늘었으므로 발전했다.
하지만 아직 8/8/8을 달성하지 못했으므로 다음 운동도 60kg 유지가 좋다.
```

## 3.3 이번 과제에서 제외할 기능

토요일에 제외하기로 한 기능도 중요합니다.

왜냐하면 제외할 것을 정하지 않으면 프로젝트가 끝없이 커지기 때문입니다.

이번 과제에서 제외한 기능은 아래입니다.

- 부상 진단
- 의학적 처방
- 소셜 로그인
- 결제
- 칼로리 분석
- 영양 분석
- 트레이너용 회원 관리
- 완전 자율 게시글 발행 Agent
- 완성형 AWS 배포

특히 부상 진단이나 의학적 처방은 하지 않습니다.

우리 서비스는:

```text
"무릎이 아픈데 계속 운동해도 될까요?"
```

에 답하는 서비스가 아닙니다.

우리 서비스는:

```text
"지난번보다 반복 수가 늘었으니,
이번에는 무게를 올릴지 유지할지 추천해줄게요."
```

에 가까운 서비스입니다.

# 4. ERD 초안 확정

## 4.1 ERD가 뭔가?

ERD는 `Entity Relationship Diagram`의 줄임말입니다.

한국어로 풀면:

```text
데이터끼리의 관계를 그린 그림
```

여기서 `Entity`는 DB 테이블이라고 생각하면 됩니다.

예를 들어 우리 서비스에는 이런 데이터가 있습니다.

- 사용자
- 게시글
- 운동
- 세트
- 댓글
- 태그
- AI 분석 결과

이 데이터들은 서로 따로 떨어져 있지 않습니다.

예를 들어:

```text
사용자 1명은 게시글 여러 개를 작성할 수 있다.
게시글 1개에는 운동 여러 개가 들어갈 수 있다.
운동 1개에는 세트 여러 개가 들어갈 수 있다.
게시글 1개에는 댓글 여러 개가 달릴 수 있다.
```

이런 관계를 정리하는 것이 ERD입니다.

## 4.2 우리 프로젝트 ERD 정답

토요일에 정한 전체 ERD 초안은 아래입니다.

```text
User
  1:N WorkoutPost
  1:N Comment
  1:N AiAnalysis

WorkoutPost
  N:1 User
  1:N WorkoutExercise
  1:N Comment
  N:M Tag through PostTag
  1:N AiAnalysis
  1:N PostEmbedding

WorkoutExercise
  N:1 WorkoutPost
  1:N WorkoutSet

WorkoutSet
  N:1 WorkoutExercise

Tag
  N:M WorkoutPost through PostTag

AiAnalysis
  N:1 User
  N:1 WorkoutPost

PostEmbedding
  N:1 User
  N:1 WorkoutPost

AgentRun
  1:N AgentStep
  1:N McpToolLog
```

처음 보면 복잡합니다.

하나씩 풀면 이렇습니다.

## 4.3 User

`User`는 사용자입니다.

회원가입하면 User 데이터가 생깁니다.

예:

```text
id: 1
email: demo@example.com
nickname: 운동초보
passwordHash: 암호화된 비밀번호
```

User는 여러 게시글을 쓸 수 있습니다.

```text
User 1명
  ├─ WorkoutPost 1
  ├─ WorkoutPost 2
  └─ WorkoutPost 3
```

그래서 관계는:

```text
User 1 : WorkoutPost N
```

입니다.

## 4.4 WorkoutPost

`WorkoutPost`는 운동 기록 게시글입니다.

예:

```text
제목: 오늘 가슴 운동
내용: 마지막 세트가 힘들었다.
작성자: 운동초보
```

하지만 운동 기록 게시판은 일반 게시판과 다릅니다.

게시글 안에 운동 표가 들어갑니다.

그래서 WorkoutPost 아래에 `WorkoutExercise`가 붙습니다.

```text
WorkoutPost
  ├─ WorkoutExercise: 벤치프레스
  └─ WorkoutExercise: 인클라인 덤벨프레스
```

## 4.5 WorkoutExercise

`WorkoutExercise`는 게시글 안에 들어가는 운동 하나입니다.

예:

```text
운동명: 벤치프레스
운동 부위: 가슴
```

한 게시글에는 여러 운동이 들어갈 수 있습니다.

예:

```text
오늘 가슴 운동 게시글
  ├─ 벤치프레스
  ├─ 인클라인 덤벨프레스
  └─ 케이블 플라이
```

그래서 관계는:

```text
WorkoutPost 1 : WorkoutExercise N
```

입니다.

## 4.6 WorkoutSet

`WorkoutSet`은 운동 하나의 세트 기록입니다.

예를 들어 벤치프레스를 3세트 했다면:

```text
벤치프레스
  ├─ 1세트: 60kg 8회
  ├─ 2세트: 60kg 8회
  └─ 3세트: 60kg 7회
```

여기서 각각의 세트가 `WorkoutSet`입니다.

그래서 관계는:

```text
WorkoutExercise 1 : WorkoutSet N
```

입니다.

## 4.7 Comment

`Comment`는 댓글입니다.

예:

```text
좋아요. 다음에는 8/8/8 도전해도 될 듯!
```

댓글은 사용자와 게시글에 연결됩니다.

```text
User 1명은 댓글 여러 개를 쓸 수 있다.
WorkoutPost 1개에는 댓글 여러 개가 달릴 수 있다.
```

## 4.8 Tag와 PostTag

`Tag`는 게시글 분류입니다.

예:

```text
가슴
벤치프레스
증량도전
하체
등
```

게시글 하나에는 태그 여러 개가 붙을 수 있습니다.

또 태그 하나는 여러 게시글에 붙을 수 있습니다.

예:

```text
"가슴" 태그
  ├─ 오늘 가슴 운동
  ├─ 벤치프레스 60kg 성공
  └─ 인클라인 덤벨 기록
```

이런 관계를 `N:M`, 즉 다대다 관계라고 합니다.

DB에서는 다대다 관계를 직접 저장하기 어렵기 때문에 중간 테이블을 둡니다.

그 중간 테이블이 `PostTag`입니다.

```text
WorkoutPost N : M Tag
       중간 연결 테이블: PostTag
```

## 4.9 AiAnalysis

`AiAnalysis`는 AI 분석 결과입니다.

게시글마다 AI 분석 결과를 저장합니다.

예:

```text
요약:
벤치프레스 총 반복 수가 지난번보다 증가했습니다.

추천:
다음 운동에서는 60kg 8/8/8을 목표로 하세요.

판단:
아직 증량보다는 반복 수 완성이 우선입니다.
```

AI 응답을 매번 새로 생성하면 비용이 계속 듭니다.

그래서 한 번 분석한 결과는 DB에 저장해두는 것이 좋습니다.

## 4.10 PostEmbedding

`PostEmbedding`은 RAG를 위한 벡터 데이터입니다.

초보자 입장에서는 이렇게 이해하면 됩니다.

```text
게시글 내용을 AI 검색용 숫자 배열로 바꿔 저장한 것
```

예를 들어:

```text
"벤치프레스 60kg 8회"
```

라는 문장을 AI가 이해하기 쉬운 숫자 배열로 바꿉니다.

```text
[0.12, -0.03, 0.91, ...]
```

이 숫자 배열을 DB에 저장하면 나중에 비슷한 운동 기록을 검색할 수 있습니다.

```text
현재 기록: 벤치프레스 60kg 8/8/7
비슷한 과거 기록: 벤치프레스 60kg 8/7/6
```

이것이 RAG의 시작입니다.

## 4.11 AgentRun, AgentStep, McpToolLog

이 셋은 AI Agent와 MCP를 위한 로그입니다.

처음부터 만들 필요는 없습니다.

나중에 AI 분석 기능을 만들 때 사용합니다.

예를 들어 AI Agent가 아래 순서로 움직였다고 합시다.

```text
1. 현재 게시글 조회
2. 이전 운동 기록 검색
3. MCP tool로 운동 정보 조회
4. GPT 호출
5. 분석 결과 저장
```

이 전체 실행 1번이 `AgentRun`입니다.

각 단계가 `AgentStep`입니다.

MCP tool을 호출한 기록이 `McpToolLog`입니다.

## 4.12 처음부터 모든 테이블을 만들지 않는 이유

토요일에 중요한 결정을 하나 했습니다.

```text
처음부터 모든 테이블을 만들지 않는다.
```

이유는 단순합니다.

처음부터 댓글, 태그, AI 분석, RAG, MCP, Agent까지 모두 만들려고 하면 시작이 너무 무거워집니다.

그래서 구현 우선순위를 이렇게 정했습니다.

```text
1. User
2. WorkoutPost
3. WorkoutExercise
4. WorkoutSet
5. Comment
6. Tag / PostTag
7. AiAnalysis
8. PostEmbedding
9. AgentRun / AgentStep / McpToolLog
```

월요일에는 이 중에서 먼저 아래 4개만 구현합니다.

```text
User
WorkoutPost
WorkoutExercise
WorkoutSet
```

왜냐하면 이 4개만 있어도 운동 기록 게시글을 저장하고 조회할 수 있기 때문입니다.

# 5. API 목록 확정

## 5.1 API 목록이 뭔가?

API 목록은 프론트엔드와 백엔드가 어떤 주소로 데이터를 주고받을지 정리한 표입니다.

쉽게 말하면:

```text
프론트엔드가 백엔드에게 요청할 수 있는 메뉴판
```

입니다.

예를 들어:

```text
GET /posts
```

는 "게시글 목록 주세요"라는 요청입니다.

```text
POST /posts
```

는 "게시글 하나 새로 저장해주세요"라는 요청입니다.

API 목록에는 보통 3가지가 들어갑니다.

```text
1. HTTP Method
2. URL
3. 설명
```

## 5.2 HTTP Method가 뭔가?

HTTP Method는 요청의 목적을 나타내는 단어입니다.

자주 쓰는 것은 아래 5개입니다.

| Method | 의미 | 예시 |
| --- | --- | --- |
| GET | 조회 | 게시글 목록 보기 |
| POST | 생성 | 게시글 작성 |
| PATCH | 일부 수정 | 게시글 제목 수정 |
| PUT | 전체 교체 | 게시글 전체 교체 |
| DELETE | 삭제 | 게시글 삭제 |

우리 프로젝트에서는 주로 `GET`, `POST`, `PATCH`, `DELETE`를 씁니다.

## 5.3 Auth API

Auth는 인증 기능입니다.

인증은 "너 누구야?"를 확인하는 기능입니다.

| Method | URL | 설명 |
| --- | --- | --- |
| POST | `/auth/signup` | 회원가입 |
| POST | `/auth/login` | 로그인 |
| GET | `/auth/me` | 내 정보 조회 |

토요일에 `/auth/refresh`는 MVP에서 제외하기로 했습니다.

처음에는 access token 하나만 사용합니다.

이유:

```text
refresh token까지 넣으면 인증 구조가 복잡해진다.
이번 과제에서는 JWT 인증의 기본 흐름을 보여주는 것이 우선이다.
```

## 5.4 Posts API

Posts는 운동 기록 게시글 기능입니다.

| Method | URL | 설명 |
| --- | --- | --- |
| GET | `/posts` | 게시글 목록 조회, 검색, 페이징 |
| POST | `/posts` | 게시글 작성 |
| GET | `/posts/:id` | 게시글 상세 조회 |
| PATCH | `/posts/:id` | 게시글 수정 |
| DELETE | `/posts/:id` | 게시글 삭제 |
| GET | `/me/posts` | 내 운동 기록 목록 |

여기서 `:id`는 URL param입니다.

예를 들어:

```text
GET /posts/3
```

이면 3번 게시글을 조회하겠다는 뜻입니다.

## 5.5 Comments API

댓글 기능입니다.

| Method | URL | 설명 |
| --- | --- | --- |
| GET | `/posts/:postId/comments` | 특정 게시글 댓글 목록 |
| POST | `/posts/:postId/comments` | 특정 게시글에 댓글 작성 |
| DELETE | `/comments/:id` | 댓글 삭제 |

댓글 작성은 특정 게시글에 붙어야 합니다.

그래서 주소에 `postId`가 들어갑니다.

```text
POST /posts/10/comments
```

뜻:

```text
10번 게시글에 댓글을 작성한다.
```

## 5.6 Tags API

태그 기능입니다.

| Method | URL | 설명 |
| --- | --- | --- |
| GET | `/tags` | 태그 목록 |
| GET | `/tags/:name/posts` | 특정 태그가 붙은 게시글 목록 |

예:

```text
GET /tags/벤치프레스/posts
```

뜻:

```text
"벤치프레스" 태그가 붙은 게시글 목록을 조회한다.
```

## 5.7 AI API

AI API는 React가 직접 FastAPI를 호출하지 않도록 설계했습니다.

프론트엔드는 NestJS만 호출합니다.

```text
React
  ↓
NestJS
  ↓
FastAPI
  ↓
OpenAI
```

React가 호출하는 API는 아래입니다.

| Method | URL | 설명 |
| --- | --- | --- |
| POST | `/ai/posts/:postId/analyze` | AI 분석 실행 |
| GET | `/ai/posts/:postId/analysis` | AI 분석 결과 조회 |

NestJS가 내부적으로 FastAPI를 호출하는 API는 아래입니다.

| Method | URL | 설명 |
| --- | --- | --- |
| GET | `/internal/health` | AI 서버 상태 확인 |
| POST | `/internal/analyze-post` | Agent 기반 운동 기록 분석 |
| POST | `/internal/index-post` | RAG용 embedding 저장 |

FastAPI 주소 앞에 `/internal`을 붙인 이유는:

```text
사용자가 직접 호출하는 API가 아니라,
NestJS가 내부적으로 호출하는 API라는 의미를 주기 위해서
```

입니다.

# 6. 프로젝트 폴더 구조 확정

## 6.1 폴더 구조를 왜 먼저 정해야 하나?

프로젝트 폴더 구조는 "각 코드가 어디에 살지"를 정하는 일입니다.

폴더 구조가 없으면 파일이 아무 곳에나 생깁니다.

그러면 시간이 지날수록 프로젝트를 이해하기 어려워집니다.

토요일에 정한 폴더 구조는 아래입니다.

```text
ai-workout-board/
  frontend/       React 화면
  backend/        NestJS 백엔드
  ai-server/      FastAPI AI 서버
  docs/           기획, 명세, 로드맵, 발표 문서
  docker-compose.yml
  .env.example
```

## 6.2 frontend 폴더

`frontend/`는 사용자가 브라우저에서 보는 화면을 담당합니다.

여기에는 React 코드가 들어갑니다.

예:

```text
frontend/
  src/
    App.tsx
    main.tsx
    pages/
      PostListPage.tsx
      PostCreatePage.tsx
```

사용자는 React 화면에서 버튼을 누릅니다.

React는 NestJS API를 호출합니다.

```text
브라우저 화면
  ↓
React
  ↓
NestJS API 호출
```

## 6.3 backend 폴더

`backend/`는 일반 백엔드 서버입니다.

여기에는 NestJS 코드가 들어갑니다.

담당 기능:

- 회원가입
- 로그인
- JWT 인증
- 게시글 CRUD
- 댓글
- 태그
- 검색
- 페이징
- DB 저장/조회
- FastAPI 호출

즉, 백엔드의 중심은 NestJS입니다.

## 6.4 ai-server 폴더

`ai-server/`는 AI 전용 서버입니다.

여기에는 FastAPI 코드가 들어갑니다.

담당 기능:

- OpenAI API 호출
- embedding 생성
- RAG 검색
- MCP tool 실행
- Agent workflow 실행
- AI 분석 결과 생성

FastAPI는 사용자가 직접 호출하지 않습니다.

NestJS가 내부적으로 호출합니다.

## 6.5 docs 폴더

`docs/`는 문서 폴더입니다.

여기에는 기획, 명세, 로드맵, 발표 준비 문서가 들어갑니다.

예:

```text
docs/
  운동기록기획.md
  운동기록명세.md
  구현로드맵.md
  2026-06-06/
    실행결정.md
    작업이슈.md
    타임테이블.md
```

이 문서들이 중요한 이유는 개발 중에 길을 잃지 않게 해주기 때문입니다.

## 6.6 docker-compose.yml

`docker-compose.yml`은 여러 프로그램을 한 번에 실행하기 위한 설정 파일입니다.

토요일에는 우선 DB만 Docker로 띄우기로 했습니다.

```text
처음:
PostgreSQL + pgvector만 Docker로 실행

나중:
backend, frontend, ai-server까지 Docker에 넣을 수 있음
```

처음부터 모든 서버를 Docker에 넣지 않는 이유는:

```text
초반에는 서버가 어떻게 실행되는지 직접 보는 게 학습에 더 좋기 때문
```

입니다.

# 7. GitHub Issue 생성

## 7.1 GitHub Issue가 뭔가?

GitHub Issue는 작업 단위를 기록하는 곳입니다.

쉽게 말하면:

```text
프로젝트 할 일 카드
```

입니다.

예를 들어:

```text
Issue #2: NestJS 백엔드 기본 세팅
Issue #3: 운동 기록 게시글 DB 모델 구현
Issue #4: 운동 기록 게시글 CRUD API 구현
```

이렇게 작업을 나누면 지금 무엇을 해야 하는지 명확해집니다.

## 7.2 왜 이슈를 만들어야 하나?

이슈를 만들지 않으면 작업이 머릿속에만 있습니다.

그러면 이런 일이 생깁니다.

- 오늘 뭘 해야 하는지 매번 다시 생각해야 한다.
- 작업 범위가 계속 커진다.
- 무엇을 완료했는지 알기 어렵다.
- PR을 만들 때 설명하기 어렵다.
- 발표 때 개발 과정을 설명하기 어렵다.

이슈를 만들면 반대로 이런 장점이 생깁니다.

- 작업 단위가 보인다.
- 우선순위가 보인다.
- 완료 기준이 생긴다.
- PR과 연결할 수 있다.
- 나중에 회고하기 쉽다.

## 7.3 토요일에 정한 이슈 목록

토요일에는 아래 이슈 흐름을 만들었습니다.

```text
#1 프로젝트 초기 세팅
#2 NestJS 백엔드 기본 세팅
#3 운동 기록 게시글 DB 모델 구현
#4 운동 기록 게시글 CRUD API 구현
#5 회원가입, 로그인, JWT 인증 구현
#6 React 게시판 화면 연결
#7 댓글, 태그, 검색, 페이징 구현
#8 FastAPI AI 서버 기본 세팅
#9 AI 운동 기록 분석 기능 구현
#10 RAG 기능 구현
#11 MCP tool 구현
#12 Agent workflow 구현
#13 README와 발표 준비
```

이슈는 기능 순서대로 쪼개는 것이 좋습니다.

처음부터 AI Agent를 만들면 안 됩니다.

먼저 일반 게시판 흐름이 있어야 합니다.

```text
DB 저장/조회
  ↓
인증
  ↓
프론트 연결
  ↓
댓글/태그/검색
  ↓
AI 서버
  ↓
AI 분석
  ↓
RAG
  ↓
MCP
  ↓
Agent
```

이 순서가 중요한 이유는 뒤 기능이 앞 기능에 의존하기 때문입니다.

예를 들어 RAG는 과거 게시글이 DB에 저장되어 있어야 가능합니다.

AI 분석은 현재 게시글과 이전 게시글을 조회할 수 있어야 가능합니다.

Agent는 RAG, MCP, GPT 호출을 순서대로 묶는 기능이므로 가장 나중에 가야 합니다.

# 8. 브랜치 전략 정하기

## 8.1 브랜치가 뭔가?

브랜치는 Git에서 작업 흐름을 나누는 기능입니다.

쉽게 말하면:

```text
main 코드를 망가뜨리지 않고 따로 작업하는 공간
```

입니다.

예를 들어 `main` 브랜치가 안정된 코드라고 합시다.

새 기능을 만들 때 바로 `main`에서 작업하면 실수했을 때 안정된 코드가 깨질 수 있습니다.

그래서 새 브랜치를 만듭니다.

```bash
git checkout -b feat/backend-posts
```

그러면 `feat/backend-posts`라는 별도 작업 공간에서 코드를 수정할 수 있습니다.

작업이 끝나면 PR을 만들고 main에 합칩니다.

## 8.2 토요일에 정한 브랜치 전략

토요일에 정한 브랜치 이름은 아래와 같습니다.

```text
main
feat/init
feat/backend-posts
feat/auth
feat/frontend-posts
feat/comments-tags-search
feat/ai-server
feat/rag
feat/mcp-agent
docs/readme
```

각 브랜치의 의미는 아래입니다.

| 브랜치 | 의미 |
| --- | --- |
| `main` | 안정된 최종 코드 |
| `feat/init` | 프로젝트 초기 세팅 |
| `feat/backend-posts` | 게시글 백엔드 기능 |
| `feat/auth` | 회원가입, 로그인, JWT |
| `feat/frontend-posts` | React 게시판 화면 |
| `feat/comments-tags-search` | 댓글, 태그, 검색, 페이징 |
| `feat/ai-server` | FastAPI 서버 |
| `feat/rag` | RAG 검색 |
| `feat/mcp-agent` | MCP와 Agent |
| `docs/readme` | README와 발표 문서 |

## 8.3 혼자 해도 브랜치를 나누는 이유

혼자 개발해도 브랜치를 나누는 것이 좋습니다.

이유:

- 작업 단위가 분리된다.
- 실수했을 때 되돌리기 쉽다.
- GitHub PR 연습이 된다.
- 나만무 팀 프로젝트 전에 협업 흐름을 연습할 수 있다.
- 발표 때 개발 과정을 설명하기 좋다.

혼자 개발한다고 해서 모든 코드를 main에 바로 넣으면 나중에 흐름이 흐려집니다.

이번 과제는 학습 목적도 있으므로 브랜치와 PR 흐름을 연습하는 것이 좋습니다.

## 8.4 기본 작업 흐름

하나의 이슈를 작업할 때 흐름은 아래입니다.

```text
1. GitHub Issue 확인
2. 이슈 번호에 맞는 브랜치 생성
3. 코드 작성
4. 로컬 테스트
5. commit
6. push
7. PR 생성
8. main에 merge
```

예를 들어 Issue #2를 작업한다면:

```bash
git checkout main
git pull origin main
git checkout -b feat/2-backend-setup
```

작업 후:

```bash
git status
git add .
git commit -m "feat: NestJS 백엔드 기본 세팅"
git push origin feat/2-backend-setup
```

PR 설명에는 보통 이렇게 씁니다.

```md
## 작업 내용

- NestJS 프로젝트를 생성했습니다.
- 환경 변수 설정을 추가했습니다.
- health check API를 추가했습니다.

## 확인 방법

- `npm run start:dev`
- `GET /health`

Closes #2
```

`Closes #2`를 쓰면 PR이 main에 merge될 때 GitHub Issue #2가 자동으로 닫힙니다.

# 9. Docker Compose 방향 정하기

## 9.1 Docker가 뭔가?

Docker는 프로그램 실행 환경을 통째로 묶어서 실행할 수 있게 해주는 도구입니다.

초보자 입장에서는 이렇게 이해하면 됩니다.

```text
"내 컴퓨터에 직접 PostgreSQL을 설치하지 않고,
PostgreSQL이 들어있는 작은 실행 상자를 띄우는 도구"
```

입니다.

이 작은 실행 상자를 컨테이너라고 부릅니다.

## 9.2 Docker Compose가 뭔가?

Docker Compose는 여러 컨테이너를 한 번에 실행하는 설정입니다.

우리 프로젝트에서는 처음에 PostgreSQL만 Docker로 실행합니다.

나중에는 아래처럼 확장할 수 있습니다.

```text
frontend 컨테이너
backend 컨테이너
ai-server 컨테이너
postgres 컨테이너
```

하지만 토요일 결정은 이랬습니다.

```text
처음에는 PostgreSQL + pgvector만 Docker로 실행한다.
```

## 9.3 왜 DB만 Docker로 띄우나?

이유는 3가지입니다.

첫째, PostgreSQL을 로컬에 직접 설치하면 환경 차이가 생길 수 있습니다.

둘째, pgvector 확장까지 직접 설치하면 초반 난이도가 올라갑니다.

셋째, NestJS와 FastAPI와 React는 처음에는 로컬에서 직접 실행하는 편이 동작 원리를 이해하기 쉽습니다.

그래서 초반 구조는 이렇게 갑니다.

```text
React: 로컬에서 npm run dev
NestJS: 로컬에서 npm run start:dev
FastAPI: 로컬에서 uvicorn 실행
PostgreSQL: Docker Compose로 실행
```

## 9.4 Docker Compose 초안

토요일에 정한 Docker Compose 방향은 아래와 같습니다.

```yaml
services:
  postgres:
    image: pgvector/pgvector:pg16
    container_name: workout-board-postgres
    restart: unless-stopped
    ports:
      - "5432:5432"
    environment:
      POSTGRES_USER: workout_user
      POSTGRES_PASSWORD: workout_password
      POSTGRES_DB: workout_board
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

이 설정의 의미는 아래입니다.

| 항목 | 의미 |
| --- | --- |
| `image` | 어떤 Docker 이미지를 사용할지 |
| `pgvector/pgvector:pg16` | pgvector가 포함된 PostgreSQL 16 이미지 |
| `container_name` | 컨테이너 이름 |
| `ports` | 내 컴퓨터와 컨테이너 포트 연결 |
| `5432:5432` | 내 컴퓨터 5432번 포트로 DB 접근 가능 |
| `environment` | DB 아이디, 비밀번호, DB 이름 |
| `volumes` | DB 데이터를 컨테이너 삭제 후에도 보존 |

## 9.5 DB 접속 문자열

NestJS에서 Prisma가 PostgreSQL에 접속하려면 DB 주소가 필요합니다.

그 주소를 `DATABASE_URL`이라고 둡니다.

```env
DATABASE_URL="postgresql://workout_user:workout_password@localhost:5432/workout_board?schema=public"
```

이 주소를 해석하면:

```text
postgresql://
  PostgreSQL DB에 접속한다.

workout_user
  DB 사용자 이름

workout_password
  DB 비밀번호

localhost
  내 컴퓨터에서 실행 중인 DB

5432
  PostgreSQL 기본 포트

workout_board
  사용할 DB 이름

schema=public
  PostgreSQL의 public schema 사용
```

## 9.6 실행 명령어

DB 컨테이너 실행:

```bash
docker compose up -d
```

실행 중인지 확인:

```bash
docker compose ps
```

로그 확인:

```bash
docker compose logs postgres
```

DB 컨테이너 중지:

```bash
docker compose down
```

데이터까지 완전히 삭제:

```bash
docker compose down -v
```

단, `-v`는 DB 데이터까지 지우는 명령이므로 조심해야 합니다.

# 10. 가능하면 진행할 작업

토요일 목표는 설계 확정이었지만, 시간이 되면 아래 작업도 진행할 수 있었습니다.

```text
backend/ 폴더 생성
ai-server/ 폴더 생성
NestJS 프로젝트 생성
FastAPI 프로젝트 생성
PostgreSQL/pgvector Docker Compose 초안 작성
```

각각이 무슨 의미인지 정리하면 아래와 같습니다.

## 10.1 backend 폴더 생성

`backend/` 폴더는 NestJS 프로젝트가 들어갈 자리입니다.

여기에 백엔드 코드가 들어갑니다.

```text
backend/
  src/
    main.ts
    app.module.ts
    app.controller.ts
```

## 10.2 ai-server 폴더 생성

`ai-server/` 폴더는 FastAPI 프로젝트가 들어갈 자리입니다.

여기에 AI 기능 코드가 들어갑니다.

```text
ai-server/
  app/
    main.py
```

## 10.3 NestJS 프로젝트 생성

NestJS는 백엔드 프레임워크입니다.

역할:

```text
프론트엔드 요청을 받고,
알맞은 함수로 보내고,
DB 작업을 하고,
응답을 돌려주는 서버 구조를 제공한다.
```

예:

```text
GET /health
  ↓
NestJS가 요청 받음
  ↓
health check 함수 실행
  ↓
JSON 응답 반환
```

## 10.4 FastAPI 프로젝트 생성

FastAPI는 Python 백엔드 프레임워크입니다.

우리 프로젝트에서는 일반 백엔드가 아니라 AI 전용 서버로 씁니다.

역할:

```text
GPT 호출
embedding 생성
RAG 검색
MCP tool 실행
Agent workflow 실행
```

## 10.5 PostgreSQL/pgvector Docker Compose 초안 작성

PostgreSQL은 일반 데이터를 저장하는 DB입니다.

pgvector는 PostgreSQL에 벡터 검색 기능을 추가하는 확장입니다.

우리 프로젝트에서 pgvector는 나중에 RAG를 위해 사용합니다.

```text
게시글 내용
  ↓
embedding 숫자 배열로 변환
  ↓
pgvector에 저장
  ↓
비슷한 운동 기록 검색
```

# 11. 토요일 완료 기준 해석

토요일 완료 기준은 아래였습니다.

```text
MVP에 포함할 기능과 제외할 기능이 정리되어 있다.
ERD와 API 목록의 초안이 있다.
프로젝트 폴더 구조가 정리되어 있다.
이후 작업할 GitHub 이슈가 생성되어 있다.
월요일부터 "뭘 만들지" 고민하지 않고 바로 구현을 시작할 수 있다.
```

하나씩 해석하면 이렇습니다.

## 11.1 MVP에 포함할 기능과 제외할 기능이 정리되어 있다

이 말은:

```text
이번 과제에서 반드시 만들 기능과,
이번 과제에서는 욕심내지 않을 기능을 구분한다.
```

라는 뜻입니다.

예를 들어 AI 분석은 필수입니다.

하지만 성장 그래프는 시간이 남으면 하는 기능입니다.

이렇게 구분해야 시간이 부족할 때 무엇을 버릴지 판단할 수 있습니다.

## 11.2 ERD와 API 목록의 초안이 있다

이 말은:

```text
DB에 어떤 데이터를 저장할지,
프론트엔드와 백엔드가 어떤 주소로 통신할지 대략 정해져 있다.
```

라는 뜻입니다.

완벽한 ERD와 API가 아니어도 됩니다.

초안이 있으면 구현하면서 수정할 수 있습니다.

초안조차 없으면 구현을 시작할 수 없습니다.

## 11.3 프로젝트 폴더 구조가 정리되어 있다

이 말은:

```text
React, NestJS, FastAPI, 문서, Docker 설정이 각각 어디에 위치할지 정해져 있다.
```

라는 뜻입니다.

## 11.4 이후 작업할 GitHub 이슈가 생성되어 있다

이 말은:

```text
월요일부터 할 일이 GitHub 작업 카드로 쪼개져 있다.
```

라는 뜻입니다.

## 11.5 월요일부터 바로 구현을 시작할 수 있다

이것이 토요일의 가장 중요한 완료 기준입니다.

토요일 작업이 잘 되면 월요일에는 이렇게 시작할 수 있습니다.

```text
오늘은 User, WorkoutPost, WorkoutExercise, WorkoutSet 모델을 만들고
게시글 작성/조회 API를 구현하면 된다.
```

즉, 월요일에 다시 서비스 방향을 고민하지 않아도 됩니다.

# 12. 토요일 결정이 월요일 구현으로 이어지는 방식

토요일 결정은 월요일 구현의 출발점입니다.

월요일 작업은 아래처럼 토요일 결정에서 바로 이어집니다.

```text
토요일 ERD:
User, WorkoutPost, WorkoutExercise, WorkoutSet 먼저 만든다.

월요일 구현:
Prisma schema에 User, WorkoutPost, WorkoutExercise, WorkoutSet 모델 작성
```

```text
토요일 API 목록:
POST /posts
GET /posts
GET /posts/:id

월요일 구현:
NestJS PostsController에서 위 API 3개 구현
```

```text
토요일 Docker Compose:
PostgreSQL + pgvector를 Docker로 실행

월요일 구현:
Prisma가 DATABASE_URL로 PostgreSQL에 연결
```

```text
토요일 폴더 구조:
backend/에 NestJS 서버
frontend/에 React 화면

월요일 구현:
backend에서 Posts API 구현
frontend에서 게시글 목록/작성 페이지 뼈대 생성
```

즉, 토요일은 설계이고 월요일은 그 설계를 코드로 옮기는 날입니다.

# 13. 초보자 기준 핵심 단어 다시 정리

## 13.1 MVP

```text
발표에서 핵심 가치를 보여줄 수 있는 최소 버전
```

우리 MVP:

```text
운동 기록을 저장하고,
AI가 이전 기록과 비교해,
다음 운동 목표를 추천한다.
```

## 13.2 ERD

```text
DB 테이블들이 서로 어떻게 연결되는지 정리한 그림
```

예:

```text
User 1명은 WorkoutPost 여러 개를 쓸 수 있다.
WorkoutPost 1개에는 WorkoutExercise 여러 개가 들어간다.
WorkoutExercise 1개에는 WorkoutSet 여러 개가 들어간다.
```

## 13.3 API

```text
프론트엔드와 백엔드가 데이터를 주고받기 위한 약속된 주소와 규칙
```

예:

```text
POST /posts
게시글을 생성한다.
```

## 13.4 URL param

```text
URL 안에 들어가는 값
```

예:

```text
GET /posts/3
```

여기서 `3`이 URL param입니다.

## 13.5 GitHub Issue

```text
작업 단위를 적어두는 GitHub 할 일 카드
```

예:

```text
Issue #4: 운동 기록 게시글 CRUD API 구현
```

## 13.6 Branch

```text
main 코드를 망가뜨리지 않고 따로 작업하는 Git 작업 공간
```

예:

```text
feat/backend-posts
```

## 13.7 Pull Request

```text
내 브랜치의 작업 내용을 main에 합쳐달라고 요청하는 것
```

혼자 해도 PR을 만들면 작업 내용을 정리하는 연습이 됩니다.

## 13.8 Docker

```text
프로그램 실행 환경을 컨테이너로 띄우는 도구
```

우리 프로젝트에서는 PostgreSQL을 Docker로 실행합니다.

## 13.9 Docker Compose

```text
여러 Docker 컨테이너 실행 방법을 한 파일에 적어두는 도구
```

현재는 DB 하나만 실행합니다.

## 13.10 pgvector

```text
PostgreSQL에서 AI embedding 벡터를 저장하고 검색할 수 있게 해주는 확장
```

RAG 기능에 필요합니다.

## 13.11 RAG

```text
AI가 답변하기 전에 DB에서 관련 자료를 찾아 참고하게 하는 방식
```

우리 서비스에서는 이전 운동 기록을 찾아 AI 분석에 넣습니다.

## 13.12 MCP

```text
AI가 외부 도구나 외부 서비스를 호출할 수 있게 해주는 연결 규칙
```

우리 서비스에서는 운동 정보 조회 tool 같은 것을 만들 수 있습니다.

## 13.13 Agent

```text
AI가 여러 단계를 순서대로 실행하며 목표를 해결하는 흐름
```

우리 서비스에서는:

```text
현재 기록 조회
이전 기록 검색
운동 정보 조회
GPT 분석
결과 저장
```

이 과정을 Agent workflow로 만들 예정입니다.

# 16. 복습 체크리스트

아래 질문에 답할 수 있으면 토요일 작업을 이해한 것입니다.

- MVP가 무엇인지 설명할 수 있는가?
- 우리 서비스의 MVP 핵심 흐름을 말할 수 있는가?
- ERD가 왜 필요한지 설명할 수 있는가?
- `User`, `WorkoutPost`, `WorkoutExercise`, `WorkoutSet`의 관계를 설명할 수 있는가?
- API 목록이 왜 필요한지 설명할 수 있는가?
- `GET /posts`와 `POST /posts`의 차이를 설명할 수 있는가?
- `GET /posts/:id`에서 `:id`가 무엇인지 설명할 수 있는가?
- `frontend`, `backend`, `ai-server` 폴더가 각각 무엇을 담당하는지 설명할 수 있는가?
- GitHub Issue가 왜 필요한지 설명할 수 있는가?
- 브랜치가 왜 필요한지 설명할 수 있는가?
- Docker Compose로 왜 DB만 먼저 띄우는지 설명할 수 있는가?

```text
  ↓
월요일에 NestJS Controller로 작성

토요일에 정한 Docker Compose
  ↓

```

이 연결만 잡히면 충분합니다.
