# 0. 이 문서를 읽는 방법

오늘 코드를 따라 치다 보면 막히는 이유는 대부분 기능이 어려워서가 아니라 **문법 모양이 낯설기 때문**이다.

예를 들어 아래 코드를 처음 보면 여러 개가 한 번에 튀어나온다.

```ts
@Injectable()
export class PostsService {
  constructor(private readonly prisma: PrismaService) {}
}
```

여기에는 다음 문법이 한꺼번에 들어 있다.

- `@Injectable()`
- `export`
- `class`
- `constructor`
- `private`
- `readonly`
- `prisma: PrismaService`
- `{}` 블록

이 문서는 월요일 작업에서 만나는 문법을 한 번에 다 이해하려고 만든 것이 아니다.

목표는 이것이다.

```text
코드를 봤을 때 "아예 무슨 글자인지 모르겠다" 상태에서
"대충 어떤 역할인지 알겠다" 상태로 가는 것
```

# 1. TypeScript 파일 기본 구조

TypeScript 파일은 보통 아래 흐름으로 생겼다.

```ts
import { Injectable } from '@nestjs/common';

@Injectable()
export class PostsService {
  constructor() {}
}
```

크게 세 부분이다.

```text
1. import
   다른 파일이나 라이브러리에서 필요한 것을 가져온다.

2. decorator
   NestJS에게 이 클래스의 역할을 알려준다.

3. class
   실제 기능을 담는 클래스를 만든다.
```

# 2. import 문법

## 2.1 import는 무엇인가?

`import`는 다른 파일이나 라이브러리에서 코드를 가져오는 문법이다.

예:

```ts
import { Injectable } from '@nestjs/common';
```

뜻:

```text
@nestjs/common이라는 라이브러리에서 Injectable이라는 것을 가져온다.
```

## 2.2 중괄호 import

```ts
import { Module } from '@nestjs/common';
```

여기서 `{ Module }`은 객체가 아니다.

이 문법은:

```text
@nestjs/common 안에 export된 여러 기능 중 Module만 꺼내온다.
```

라는 뜻이다.

여러 개를 가져올 수도 있다.

```ts
import { Injectable, OnModuleDestroy, OnModuleInit } from '@nestjs/common';
```

뜻:

```text
@nestjs/common에서 Injectable, OnModuleDestroy, OnModuleInit 세 개를 가져온다.
```

## 2.3 default import

React 쪽에서 이런 코드를 볼 수 있다.

```tsx
import AboutPage from './pages/AboutPage';
```

이건 default export된 값을 가져오는 문법이다.

중괄호가 없다.

```text
중괄호 import:
정해진 이름으로 export된 것을 가져온다.

default import:
그 파일의 대표 export를 가져온다.
```

## 2.4 경로 import

```ts
import { PrismaService } from '../prisma/prisma.service';
```

뜻:

```text
현재 파일 위치에서 한 폴더 위로 올라간 뒤,
prisma/prisma.service 파일에서 PrismaService를 가져온다.
```

경로에서 자주 보는 기호:

| 기호 | 의미 |
| --- | --- |
| `./` | 현재 폴더 |
| `../` | 한 단계 위 폴더 |
| `../../` | 두 단계 위 폴더 |

# 3. export 문법

## 3.1 export는 무엇인가?

`export`는 다른 파일에서 가져다 쓸 수 있게 공개하는 문법이다.

예:

```ts
export class PrismaService {}
```

뜻:

```text
PrismaService 클래스를 다른 파일에서 import할 수 있게 공개한다.
```

그래서 다른 파일에서 이렇게 가져올 수 있다.

```ts
import { PrismaService } from '../prisma/prisma.service';
```

## 3.2 export default

React 페이지에서 이런 코드가 나온다.

```tsx
export default function PostListPage() {
  return <main>...</main>;
}
```

뜻:

```text
이 파일의 대표 export는 PostListPage 함수다.
```

그래서 가져올 때 중괄호 없이 가져온다.

```tsx
import PostListPage from './pages/PostListPage';
```

# 4. class 문법

## 4.1 class는 무엇인가?

`class`는 객체를 만들기 위한 설계도다.

예:

```ts
export class PostsService {}
```

뜻:

```text
PostsService라는 클래스를 만든다.
```

NestJS에서는 Controller, Service, Module을 대부분 클래스로 만든다.

```ts
export class AppModule {}
export class PrismaService {}
export class PostsController {}
export class PostsService {}
```

## 4.2 class 안의 함수

class 안에 있는 함수는 보통 메서드라고 부른다.

```ts
export class PostsService {
  findAll() {
    return [];
  }
}
```

여기서 `findAll()`은 `PostsService` 클래스의 메서드다.

## 4.3 this

`this`는 현재 객체 자신을 뜻한다.

```ts
this.prisma.workoutPost.findMany()
```

뜻:

```text
현재 PostsService 객체 안에 있는 prisma를 사용한다.
```

NestJS에서 constructor로 주입받은 값은 보통 `this.이름`으로 사용한다.

# 5. constructor 문법

## 5.1 constructor는 무엇인가?

`constructor`는 클래스 객체가 만들어질 때 자동으로 실행되는 함수다.

예:

```ts
class User {
  constructor(name: string) {
    console.log(name);
  }
}

new User('철수');
```

`new User('철수')`가 실행될 때 constructor가 자동 실행된다.

## 5.2 NestJS에서 constructor는 왜 쓰나?

NestJS에서는 constructor를 주로 **의존성 주입**에 사용한다.

예:

```ts
constructor(private readonly prisma: PrismaService) {}
```

뜻:

```text
이 클래스가 만들어질 때 PrismaService를 받아서
this.prisma로 저장한다.
```

풀어 쓰면 대략 이런 느낌이다.

```ts
private readonly prisma: PrismaService;

constructor(prisma: PrismaService) {
  this.prisma = prisma;
}
```

TypeScript에서는 이걸 짧게 쓸 수 있다.

```ts
constructor(private readonly prisma: PrismaService) {}
```

## 5.3 private

```ts
private readonly prisma: PrismaService
```

`private`은 클래스 안에서만 쓰겠다는 뜻이다.

즉 `prisma`는 `PostsService` 내부에서만 쓰는 값이다.

## 5.4 readonly

`readonly`는 처음 정한 뒤 다시 바꾸지 않겠다는 뜻이다.

```ts
constructor(private readonly prisma: PrismaService) {}
```

뜻:

```text
처음 받은 PrismaService를 this.prisma에 저장하고,
나중에 다른 값으로 바꾸지 않겠다.
```

# 6. 타입 표기 문법

TypeScript는 JavaScript에 타입을 붙인 언어다.

## 6.1 변수나 값 뒤의 `: 타입`

```ts
id: number
```

뜻:

```text
id는 number 타입이다.
```

예:

```ts
findOne(id: number) {
  return this.postsService.findOne(id);
}
```

뜻:

```text
findOne 함수는 id라는 숫자를 받는다.
```

## 6.2 자주 보는 타입

| 타입 | 의미 |
| --- | --- |
| `string` | 문자열 |
| `number` | 숫자 |
| `boolean` | 참/거짓 |
| `Date` | 날짜 객체 |
| `SomeDto[]` | SomeDto 여러 개가 들어있는 배열 |

예:

```ts
title: string;
weightKg: number;
exercises: CreateWorkoutExerciseDto[];
```

## 6.3 선택값 `?`

```ts
memo?: string;
```

뜻:

```text
memo는 있어도 되고 없어도 된다.
있다면 string이어야 한다.
```

DTO에서 자주 나온다.

```ts
perceivedDifficulty?: number;
targetReps?: number;
memo?: string;
```

# 7. 데코레이터 문법

## 7.1 `@`가 붙은 것은 무엇인가?

TypeScript/NestJS에서 `@`가 붙은 것은 대부분 데코레이터다.

데코레이터는:

```text
클래스, 함수, 파라미터, 필드에 역할 설명표를 붙이는 문법
```

이다.

예:

```ts
@Controller('posts')
export class PostsController {}
```

뜻:

```text
NestJS야, 이 클래스는 /posts 요청을 담당하는 Controller야.
```

## 7.2 @Module

`@Module`은 NestJS에게 이 클래스가 **기능 묶음**이라는 것을 알려주는 설정표다.

중요한 점:

```text
@Module 자체가 클래스인 것은 아니다.
export class PostsModule {} 이 클래스이고,
@Module({...})은 그 클래스 위에 붙이는 NestJS 설정표다.
```

예:

```ts
@Module({
  controllers: [PostsController],
  providers: [PostsService],
})
export class PostsModule {}
```

뜻:

```text
PostsController와 PostsService를 PostsModule이라는 기능 묶음으로 등록한다.
```

조금 더 풀면 NestJS는 이 코드를 보고 이렇게 이해한다.

```text
아, PostsModule은 그냥 일반 TypeScript 클래스가 아니구나.
이건 NestJS 앱의 기능 묶음이구나.

이 모듈 안에는
PostsController라는 요청 담당자가 있고,
PostsService라는 실제 작업자가 있구나.
```

그래서 위 설정의 효과는 아래와 같다.

```text
1. PostsController를 요청 담당자로 등록한다.
2. PostsService를 작업자로 등록한다.
3. PostsController가 PostsService를 constructor로 받을 수 있게 연결한다.
4. 이 PostsModule을 AppModule에 import하면 /posts API가 실제로 열린다.
```

여기서 4번이 중요하다.

`PostsModule` 파일을 만들었다고 해서 NestJS 앱 전체가 자동으로 아는 것은 아니다.

NestJS 앱은 보통 `AppModule`에서 시작한다.

```ts
const app = await NestFactory.create(AppModule);
```

즉, NestJS는 처음에 `AppModule`을 읽고 앱을 시작한다.

그래서 `PostsModule`을 앱에 포함하려면 `AppModule`의 `imports`에 넣어야 한다.

```ts
@Module({
  imports: [PostsModule],
})
export class AppModule {}
```

이렇게 하면 NestJS의 흐름은 아래처럼 된다.

```text
AppModule을 읽는다.
  ↓
imports에 PostsModule이 있다.
  ↓
PostsModule도 읽는다.
  ↓
PostsModule 안의 PostsController를 발견한다.
  ↓
PostsController 안의 @Get, @Post를 읽는다.
  ↓
/posts API를 서버에 등록한다.
```

비유하면:

```text
PostsModule = /posts 기능 설명서
AppModule = 앱 전체 목차
```

기능 설명서를 써놔도 전체 목차에 넣지 않으면 앱이 그 설명서를 읽지 않는다.

그래서:

```ts
imports: [PostsModule]
```

은:

```text
AppModule아, PostsModule도 앱에 포함해서 읽어라.
```

라는 뜻이다.

만약 `@Module`이 없으면 NestJS는 아래 클래스를 그냥 일반 TypeScript 클래스로만 본다.

```ts
export class PostsModule {}
```

이 경우 NestJS 입장에서는 알 수 있는 정보가 없다.

```text
이 클래스가 무슨 역할이지?
Controller가 뭐지?
Service가 뭐지?
서버에 어떤 API를 등록해야 하지?
```

그래서 `@Module`은 NestJS에게:

```text
이 클래스 안에 어떤 요청 담당자와 작업자를 등록할지 알려주는 설정표
```

라고 이해하면 된다.

`@Module()` 안에는 객체가 들어간다.

자주 쓰는 속성:

| 속성 | 의미 |
| --- | --- |
| `imports` | 다른 모듈을 가져와 사용 |
| `controllers` | HTTP 요청 담당 클래스 등록 |
| `providers` | NestJS가 만들어 관리할 객체 등록 |
| `exports` | 다른 모듈에서도 사용할 수 있게 내보냄 |

## 7.3 @Global

```ts
@Global()
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class PrismaModule {}
```

뜻:

```text
PrismaModule을 전역 모듈처럼 사용하겠다.
```

처음 배우는 단계에서는 이렇게 이해하면 된다.

```text
PrismaService를 여러 곳에서 편하게 쓰게 하기 위한 표시
```

## 7.4 @Injectable

```ts
@Injectable()
export class PostsService {}
```

뜻:

```text
NestJS가 이 클래스를 객체로 만들어 관리하고,
필요한 곳에 주입할 수 있다.
```

Service 클래스에 자주 붙는다.

## 7.5 @Controller

```ts
@Controller('posts')
export class PostsController {}
```

뜻:

```text
/posts로 시작하는 요청은 이 클래스가 담당한다.
```

## 7.6 @Get, @Post, @Patch, @Delete

```ts
@Post()
create() {}
```

뜻:

```text
POST /posts 요청이 오면 create 함수를 실행한다.
```

왜 `/posts`냐면 클래스 위에 `@Controller('posts')`가 있기 때문이다.

```ts
@Controller('posts')
export class PostsController {
  @Post()
  create() {}
}
```

결과:

```text
POST /posts
```

상세 조회는 이렇게 생겼다.

```ts
@Get(':id')
findOne() {}
```

결과:

```text
GET /posts/1
GET /posts/2
GET /posts/3
```

## 7.7 @Body

```ts
create(@Body() createWorkoutPostDto: CreateWorkoutPostDto) {}
```

뜻:

```text
HTTP 요청 body를 꺼내서 createWorkoutPostDto에 넣는다.
```

React나 curl이 이런 JSON을 보내면:

```json
{
  "title": "오늘 가슴 운동",
  "bodyPart": "가슴"
}
```

`@Body()`가 이 데이터를 꺼낸다.

## 7.8 @Param

```ts
findOne(@Param('id', ParseIntPipe) id: number) {}
```

뜻:

```text
URL에 들어있는 id 값을 꺼낸다.
```

예:

```text
GET /posts/3
```

이면:

```ts
id = 3
```

## 7.9 ParseIntPipe

URL param은 기본적으로 문자열로 들어온다.

```text
GET /posts/3
```

여기서 `"3"`은 처음에는 문자열이다.

`ParseIntPipe`는 이것을 숫자로 바꿔준다.

```ts
@Param('id', ParseIntPipe) id: number
```

뜻:

```text
URL에서 id를 꺼내고, 숫자로 변환해서 id 변수에 넣는다.
```

# 8. DTO와 class-validator 문법

## 8.1 DTO class

DTO는 요청 데이터의 모양을 정하는 클래스다.

```ts
export class CreateWorkoutPostDto {
  title: string;
  workoutDate: string;
  bodyPart: string;
}
```

뜻:

```text
게시글 작성 요청에는 title, workoutDate, bodyPart가 들어온다.
```

## 8.2 검증 데코레이터

DTO 필드 위에 붙는 데코레이터는 요청값 검증 규칙이다.

```ts
@IsString()
@IsNotEmpty()
title: string;
```

뜻:

```text
title은 문자열이어야 하고 비어 있으면 안 된다.
```

## 8.3 오늘 쓰는 검증 데코레이터

| 데코레이터 | 의미 |
| --- | --- |
| `@IsString()` | 문자열이어야 함 |
| `@IsNotEmpty()` | 비어 있으면 안 됨 |
| `@IsInt()` | 정수여야 함 |
| `@IsNumber()` | 숫자여야 함 |
| `@Min(1)` | 최소값 1 |
| `@Max(10)` | 최대값 10 |
| `@IsOptional()` | 없어도 됨 |
| `@IsDateString()` | 날짜 문자열이어야 함 |
| `@IsArray()` | 배열이어야 함 |
| `@ArrayMinSize(1)` | 배열에 최소 1개 이상 있어야 함 |
| `@ValidateNested({ each: true })` | 배열 안 객체들도 검증함 |
| `@Type(() => SomeDto)` | 안쪽 객체를 SomeDto로 변환함 |

## 8.4 DTO 예시 읽기

```ts
export class CreateWorkoutSetDto {
  @IsInt()
  @Min(1)
  setNumber: number;

  @IsInt()
  @Min(0)
  reps: number;

  @IsOptional()
  @IsInt()
  @Min(1)
  @Max(10)
  perceivedDifficulty?: number;
}
```

읽으면 이렇게 된다.

```text
setNumber:
정수여야 하고 1 이상이어야 한다.

reps:
정수여야 하고 0 이상이어야 한다.

perceivedDifficulty:
없어도 된다.
있다면 정수여야 하고 1 이상 10 이하이어야 한다.
```

## 8.5 배열 DTO

```ts
@IsArray()
@ArrayMinSize(1)
@ValidateNested({ each: true })
@Type(() => CreateWorkoutSetDto)
sets: CreateWorkoutSetDto[];
```

뜻:

```text
sets는 배열이다.
배열 안에는 최소 1개 이상 있어야 한다.
배열 안의 각 객체도 CreateWorkoutSetDto 규칙으로 검사한다.
```

# 9. async / await 문법

## 9.1 왜 async/await이 필요한가?

처음에는 이렇게 생각하기 쉽다.

```text
컴퓨터는 코드를 위에서 아래로 순서대로 실행하니까,
DB 조회도 알아서 끝날 때까지 기다리지 않을까?
```

하지만 JavaScript/TypeScript에서는 DB 조회, 파일 읽기, 외부 API 호출처럼 시간이 오래 걸리는 작업을 만났을 때 **무조건 그 자리에서 멈춰 기다리지 않는다.**

예를 들어 아래 코드를 보자.

```ts
const posts = this.prisma.workoutPost.findMany();
console.log(posts);
```

겉으로 보면 `posts`에 게시글 목록이 들어올 것 같지만, 실제로는 바로 게시글 목록이 들어오지 않는다.

DB 조회는 시간이 걸리기 때문에 `posts`에는 실제 결과가 아니라 **Promise**가 들어온다.

```text
posts = 게시글 목록
```

이 아니라:

```text
posts = "나중에 게시글 목록을 줄게"라는 Promise
```

가 되는 것이다.

그래서 진짜 결과가 필요할 때는 `await`로 기다려야 한다.

```ts
const posts = await this.prisma.workoutPost.findMany();
console.log(posts);
```

뜻:

```text
DB 조회가 끝날 때까지 기다린 뒤,
진짜 게시글 목록을 posts에 넣는다.
```

## 9.2 Promise는 무엇인가?

`Promise`는 **지금 당장은 결과가 없지만, 나중에 결과를 줄 작업**을 담는 객체다.

예를 들어 식당에서 음식을 주문했다고 생각하면 된다.

```text
주문표 = Promise
음식 = 실제 결과
```

주문표를 받았다고 음식이 손에 있는 것은 아니다.

음식이 나올 때까지 기다려야 실제 음식을 받을 수 있다.

코드에서도 비슷하다.

```ts
const result = this.prisma.workoutPost.findMany();
```

이때 `result`는 게시글 목록이 아니라 Promise다.

```text
result = "나중에 게시글 목록을 줄게"라는 약속
```

진짜 게시글 목록을 받으려면:

```ts
const result = await this.prisma.workoutPost.findMany();
```

처럼 써야 한다.

## 9.3 await는 무엇인가?

`await`는 Promise가 끝날 때까지 기다리는 문법이다.

```ts
const author = await this.getDemoUser();
```

뜻:

```text
getDemoUser()가 끝날 때까지 기다린다.
끝나면 그 결과를 author에 넣는다.
```

DB 저장도 마찬가지다.

```ts
const post = await this.prisma.workoutPost.create({
  data: {
    title: '오늘 운동',
  },
});
```

뜻:

```text
DB에 게시글 저장 요청을 보낸다.
저장이 끝날 때까지 기다린다.
저장된 게시글 결과를 post에 넣는다.
```

## 9.4 async는 무엇인가?

`async`는 이 함수 안에서 `await`를 사용할 수 있게 해주는 표시다.

```ts
async create(createWorkoutPostDto: CreateWorkoutPostDto) {
  const author = await this.getDemoUser();
}
```

뜻:

```text
create 함수 안에서는 시간이 걸리는 작업을 await로 기다릴 수 있다.
```

`await`를 쓰려면 보통 함수 앞에 `async`가 있어야 한다.

## 9.5 async와 await는 보통 같이 나온다

```ts
async create(createWorkoutPostDto: CreateWorkoutPostDto) {
  const author = await this.getDemoUser();

  return this.prisma.workoutPost.create({
    data: {
      authorId: author.id,
      title: createWorkoutPostDto.title,
    },
  });
}
```

읽는 순서는 아래와 같다.

```text
1. create 함수는 async 함수다.
2. getDemoUser()는 시간이 걸릴 수 있다.
3. await로 getDemoUser() 결과를 기다린다.
4. 결과를 author에 넣는다.
5. author.id를 사용해서 게시글을 저장한다.
```

## 9.6 핵심 정리

```text
Promise:
나중에 결과를 줄 작업

await:
Promise가 끝날 때까지 기다리고 진짜 결과를 꺼낸다.

async:
함수 안에서 await를 쓸 수 있게 표시한다.
```

한 문장으로 정리하면:

```text
컴퓨터가 알아서 기다리는 것이 아니라,
JavaScript는 시간이 걸리는 작업을 Promise로 넘기기 때문에
"여기서는 진짜 결과가 필요하니 기다려라"라고 await를 써야 한다.
```

[

# 12. `??` 문법

```ts
orderIndex: exercise.orderIndex ?? index,
```

`??`는 nullish coalescing 연산자다.

처음에는 이렇게 이해하면 된다.

```text
왼쪽 값이 null 또는 undefined이면 오른쪽 값을 사용한다.
```

즉:

```ts
exercise.orderIndex ?? index
```

뜻:

```text
exercise.orderIndex가 있으면 그 값을 쓰고,
없으면 index를 쓴다.
```

예:

```ts
undefined ?? 0  // 결과 0
3 ?? 0          // 결과 3
```

# 13. `as const` 문법

월요일 문서에 이런 코드가 나온다.

```ts
orderBy: {
  orderIndex: 'asc' as const,
},
```

여기서 `as const`는 TypeScript에게:

```text
이 값은 그냥 string이 아니라 정확히 'asc'라는 값이다.
```

라고 알려주는 문법이다.

Prisma는 `orderBy`에서 `'asc'` 또는 `'desc'` 같은 정해진 값만 받는다.

그런데 TypeScript가 `'asc'`를 넓게 `string`으로 추론하면 타입 오류가 날 수 있다.

그래서:

```ts
'asc' as const
```

로 정확히 고정해주는 것이다.

# 17. NestJS Controller 문법

월요일 문서의 Controller는 아래처럼 생겼다.

```ts
@Controller('posts')
export class PostsController {
  constructor(private readonly postsService: PostsService) {}

  @Post()
  create(@Body() createWorkoutPostDto: CreateWorkoutPostDto) {
    return this.postsService.create(createWorkoutPostDto);
  }

  @Get()
  findAll() {
    return this.postsService.findAll();
  }

  @Get(':id')
  findOne(@Param('id', ParseIntPipe) id: number) {
    return this.postsService.findOne(id);
  }
}
```

읽는 순서:

```text
1. @Controller('posts')
   /posts 요청 묶음이다.

2. constructor(...)
   PostsService를 받아서 this.postsService로 저장한다.

3. @Post()
   POST /posts 요청을 create 함수에 연결한다.

4. @Body()
   요청 body를 꺼낸다.

5. @Get()
   GET /posts 요청을 findAll 함수에 연결한다.

6. @Get(':id')
   GET /posts/:id 요청을 findOne 함수에 연결한다.

7. @Param('id', ParseIntPipe)
   URL의 id 값을 숫자로 바꿔 꺼낸다.
```

뜻:

```text
PostListPage라는 화면 컴포넌트를 만든다.
```

React 컴포넌트는 보통 화면을 return한다.

## 20.2 JSX 태그

```tsx
<main>
  <h1>운동 기록 게시판</h1>
</main>
```

HTML처럼 보이지만 실제로는 JavaScript 안에서 쓰는 JSX 문법이다.

## 20.3 Fragment

```tsx
return (
  <>
    <header>...</header>
    <Routes>...</Routes>
  </>
);
```

`<>...</>`는 Fragment다.

뜻:

```text
여러 태그를 하나로 감싸되, 실제 DOM에는 불필요한 태그를 만들지 않는다.
```

React 컴포넌트는 보통 하나의 부모 요소만 return해야 하므로 Fragment를 자주 쓴다.

## 20.4 Link

```tsx
<Link to="/posts">운동 기록</Link>
```

뜻:

```text
/posts 경로로 이동하는 링크를 만든다.
```

일반 HTML의 `<a>`와 비슷하지만 React Router가 화면 전환을 처리한다.

## 20.5 Routes와 Route

```tsx
<Routes>
  <Route path="/posts" element={<PostListPage />} />
  <Route path="/posts/new" element={<PostCreatePage />} />
</Routes>
```

뜻:

```text
/posts 주소에서는 PostListPage를 보여준다.
/posts/new 주소에서는 PostCreatePage를 보여준다.
```

## 20.6 element={<Component />}

```tsx
element={<PostListPage />}
```

여기서 `{}`는 JavaScript 값을 JSX 안에 넣는 문법이다.

`<PostListPage />`는 React 컴포넌트다.

즉:

```text
element 속성에 PostListPage 컴포넌트를 넣는다.
```

## 20.7 {' | '}

```tsx
<Link to="/">Home</Link> {' | '}
<Link to="/posts">운동 기록</Link>
```

JSX 안에서 문자열을 JavaScript 표현식으로 넣은 것이다.

결과 화면에는:

```text
Home | 운동 기록
```

처럼 보인다.

# 21. JSON 문법

API 테스트에서 요청 body는 JSON으로 보낸다.

예:

```json
{
  "title": "오늘 가슴 운동",
  "workoutDate": "2026-06-08",
  "bodyPart": "가슴",
  "exercises": [
    {
      "exerciseName": "벤치프레스",
      "weightKg": 60,
      "sets": [
        { "setNumber": 1, "reps": 8 }
      ]
    }
  ]
}
```

JSON 규칙:

- 문자열 key는 반드시 큰따옴표를 쓴다.
- 문자열 값도 큰따옴표를 쓴다.
- 숫자는 따옴표 없이 쓴다.
- 객체는 `{}`로 감싼다.
- 배열은 `[]`로 감싼다.
- 마지막 항목 뒤에는 쉼표를 붙이지 않는다.

잘못된 JSON:

```json
{
  "title": "오늘 가슴 운동",
}
```

마지막 쉼표 때문에 에러가 날 수 있다.

# 26. 오늘 문법 핵심 요약

오늘 가장 많이 나오는 문법은 아래다.

```ts
import { Something } from 'somewhere';
```

다른 곳에서 필요한 코드를 가져온다.

```ts
export class Something {}
```

다른 파일에서 쓸 수 있게 클래스를 공개한다.

```ts
@Injectable()
```

NestJS가 이 클래스를 관리하고 주입할 수 있게 표시한다.

```ts
constructor(private readonly prisma: PrismaService) {}
```

PrismaService를 받아서 `this.prisma`로 저장한다.

```ts
async create() {
  await ...
}
```

시간 걸리는 작업을 기다린다.

```ts
@Controller('posts')
```

`/posts` 요청을 담당한다.

```ts
@Post()
```

`POST /posts` 요청과 연결한다.

```ts
@Body()
```

요청 body를 꺼낸다.

```ts
@Param('id', ParseIntPipe)
```

URL의 id 값을 꺼내 숫자로 바꾼다.

```prisma
model User {}
```

Prisma에서 DB 테이블을 정의한다.

```tsx
<Route path="/posts" element={<PostListPage />} />
```

React Router에서 `/posts` 주소에 보여줄 화면을 정한다.

# 27. 최종 한 문장

오늘 만나는 문법은 크게 세 종류다.

```text
TypeScript 문법:
class, import, export, constructor, async, await, type

NestJS 문법:
@Module, @Controller, @Injectable, @Get, @Post, @Body, @Param, providers

Prisma 문법:
model, @id, @relation, findMany, create, include, nested create
```

이 세 가지가 섞여 있어서 어렵게 느껴지는 것이다.

처음부터 전부 외우려고 하지 말고, 코드를 볼 때마다 이렇게 나누면 된다.

```text
이건 TypeScript 기본 문법인가?
이건 NestJS에게 붙이는 표시인가?
이건 Prisma가 DB를 다루는 문법인가?
```

이렇게 분리해서 보면 훨씬 덜 막힌다.
