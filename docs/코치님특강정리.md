# 1. 특강 전체 관점

코치님 특강의 큰 메시지는 두 가지입니다.

첫째, 풀스택 엔지니어가 배워야 할 범위는 매우 넓지만, 모든 기술 이름을 외우는 것이 목표는 아닙니다. 프레임워크나 언어가 달라져도 밑에 깔린 개념은 많이 겹칩니다. 예를 들어 REST API, 인증, WebSocket, SQL, 트랜잭션, 캐시, 비동기 처리, 아키텍처 같은 개념은 특정 프레임워크에만 묶이지 않습니다.

둘째, 나만무 프로젝트는 멋진 아이디어 경진대회가 아니라 개발자가 되기 위한 프로젝트입니다. 따라서 프로젝트 주제는 나중에 면접에서 기술적으로 설명할 수 있어야 합니다. “무엇을 만들었는가”보다 “어떤 기술적 문제를 어떻게 풀었는가”가 더 중요합니다.

이번 AI 게시판 과제와 나만무 프로젝트를 준비할 때도 같은 기준을 적용해야 합니다.

- 기술 이름보다 개념을 먼저 잡는다.
- 팀 상황에 맞는 스택을 선택한다.
- 선택한 스택 안에서 핵심 개념을 실제로 구현한다.
- 단순 CRUD에서 끝내지 않고 인증, 보안, DB 설계, 비동기 처리, AI 운영, 배포까지 연결한다.
- 발표장에서 안정적으로 데모할 수 있는 구조를 만든다.
- 면접에서 “왜 이 기술을 선택했는지”와 “어떤 문제를 해결했는지”를 설명할 수 있게 한다.

# 2. PDF 로드맵 재구성

PDF는 풀스택 엔지니어가 볼 기술 범위를 한 장에 펼친 로드맵입니다. 크게 다음 축으로 나뉩니다.

## 2.1 Frontend

프론트엔드는 사용자가 직접 보는 화면과 상호작용을 담당합니다.

### Framework

- HTML
- CSS
- JavaScript
- TypeScript
- React
- Vue.js

이번 과정과 과제에서는 React를 중심으로 잡는 것이 자연스럽습니다. React를 선택했다면 Vue.js까지 동시에 깊게 하려고 하기보다, React에서 핵심 개념을 제대로 경험하는 것이 좋습니다.

### Rendering

프론트엔드에서 렌더링은 화면을 언제, 어디서, 어떤 방식으로 그릴지에 대한 개념입니다.

- CSR: Client Side Rendering
- SSR: Server Side Rendering
- SSG: Static Site Generation
- ISR: Incremental Static Regeneration
- RSC: React Server Components

이번 AI 게시판에서는 처음부터 SSR, ISR, RSC를 모두 깊게 다룰 필요는 없습니다. 다만 React 화면이 상태 변화에 따라 다시 그려지는 흐름, 클라이언트에서 API를 호출해 데이터를 보여주는 흐름은 반드시 이해해야 합니다.

### State, Props, State Management

React에서 중요한 기본 개념입니다.

- Component
- Props
- State
- 전역 상태 관리
- Redux Toolkit
- Zustand
- Pinia

이번 프로젝트에서는 로그인 상태, 사용자 정보, 게시글 작성 상태, AI 추천 결과, 로딩 상태, 에러 상태를 어떻게 관리할지가 중요합니다.

처음에는 React 기본 state와 context만으로 시작해도 됩니다. 상태가 복잡해지면 Zustand나 Redux Toolkit을 검토할 수 있습니다.

### UI Framework

PDF에 나온 UI 관련 키워드는 다음과 같습니다.

- Tailwind CSS
- MUI
- shadcn
- Bootstrap

UI 프레임워크는 프로젝트 속도를 높이기 위한 도구입니다. 하지만 기능 자체보다 UI 라이브러리 선택에 시간을 너무 많이 쓰면 안 됩니다. 이번 과제에서는 게시판 목록, 글쓰기 폼, 댓글, AI 결과 패널이 명확하게 보이면 충분합니다.

### Testing

프론트엔드 테스트 범위입니다.

- Unit Test
- Integration Test
- E2E Test

시간이 부족하다면 모든 테스트를 다 넣기보다, 핵심 사용자 흐름 하나를 E2E로 확인하는 방식도 좋습니다. 예를 들어 “로그인 -> 게시글 작성 -> AI 추천 확인” 흐름을 테스트할 수 있습니다.

### Accessibility

PDF의 `a11y`는 accessibility, 즉 접근성을 뜻합니다.

접근성은 장애가 있는 사용자만을 위한 것이 아닙니다. 버튼 이름, 폼 label, 키보드 조작, 색 대비, 글자 크기 같은 기본 품질과 연결됩니다. 나중에 실제 사용자에게 서비스를 제공한다면 반드시 고려해야 합니다.

## 2.2 Backend

백엔드는 사용자의 요청을 처리하고, 인증을 검증하고, 데이터를 저장하고, 외부 서비스와 통신하는 역할을 합니다.

### Framework

PDF와 강의에서 언급된 백엔드 프레임워크입니다.

- FastAPI
- Spring Boot
- NestJS
- Next.js

이름이 비슷하지만 역할이 다릅니다.

- FastAPI: Python 기반 API 서버 프레임워크
- Spring Boot: Java 기반 백엔드 프레임워크
- NestJS: TypeScript 기반 백엔드 프레임워크
- Next.js: React 기반 풀스택/프론트엔드 프레임워크

이번 팀 회의에서는 일반 백엔드는 NestJS, AI 서버는 FastAPI로 분리하는 방향이 합리적이라고 보았습니다.

### API

백엔드를 만든다는 것은 결국 API 계약을 설계하고 구현한다는 뜻입니다.

- REST
- gRPC
- GraphQL
- Webhook
- API design
- Error handling
- HTTP 3xx, 4xx, 5xx
- CRUD

이번 과제의 최소 목표는 REST API를 제대로 설계하는 것입니다.

API 설계에서 봐야 할 것은 단순히 URL이 아닙니다.

- 리소스를 어떻게 나눌 것인가
- HTTP method를 어떻게 쓸 것인가
- 상태 코드를 어떻게 정할 것인가
- 에러 응답 형식을 어떻게 통일할 것인가
- 인증이 필요한 API와 공개 API를 어떻게 나눌 것인가
- React와 NestJS 사이의 계약을 어떻게 안정적으로 유지할 것인가
- NestJS와 FastAPI 사이의 내부 API 계약을 어떻게 만들 것인가

### Authentication

인증과 인가는 모든 백엔드 프로젝트에 들어가는 핵심 개념입니다.

- Session
- JWT
- OAuth2
- OIDC

이번 과제에서는 JWT 기반 로그인을 기본으로 잡는 것이 현실적입니다.

나중에 확장한다면 OAuth2 기반 소셜 로그인도 붙일 수 있습니다. 다만 OAuth2를 붙이기 전에 인증과 인가의 차이, access token과 refresh token의 역할, 토큰 만료와 재발급 흐름을 먼저 이해해야 합니다.

### Realtime

실시간 기능과 관련된 키워드입니다.

- WebSocket
- SSE

WebSocket은 양방향 통신입니다. 채팅, 실시간 협업, 실시간 알림처럼 클라이언트와 서버가 계속 메시지를 주고받아야 할 때 사용합니다.

SSE는 서버에서 클라이언트로 이벤트를 보내는 방식입니다. 양방향이 필요 없고 서버가 업데이트를 밀어주는 정도라면 WebSocket보다 단순합니다.

이번 과제에서는 필수는 아니지만, AI 작업 진행 상태를 보여주고 싶다면 SSE를 검토할 수 있습니다.

### Security

PDF와 강의에서 나온 보안 관련 키워드입니다.

- HTTPS
- OWASP Top 10
- CORS
- CSRF
- Rate Limit
- 인증 토큰 관리
- XSS

보안은 토이 프로젝트와 실제 서비스의 차이를 만드는 영역입니다.

이번 프로젝트에서 최소한 고려해야 할 것은 다음입니다.

- React와 NestJS가 다른 origin에서 실행될 때 CORS 설정
- JWT를 어디에 저장할지
- API key를 프론트엔드에 노출하지 않기
- AI API 남용을 막기 위한 rate limit
- 게시글/댓글 입력값에서 XSS 방어
- FastAPI를 외부에 직접 노출하지 않기

## 2.3 Data Layer

데이터 레이어는 서비스의 기억장치입니다. 게시판 프로젝트라도 데이터 모델링을 제대로 하면 면접에서 이야기할 내용이 많아집니다.

### RDBMS

- MySQL
- PostgreSQL
- SQL

강의에서는 RDBMS를 의도적으로 경험해보는 것이 좋다고 했습니다. PostgreSQL은 오픈소스 관계형 DB로 많이 선택되고, 이번 과제에서는 pgvector와도 잘 맞습니다.

### SQL

기본 SQL은 반드시 직접 써봐야 합니다.

- SELECT
- INSERT
- UPDATE
- DELETE
- WHERE
- GROUP BY
- ORDER BY
- JOIN
- INNER JOIN
- OUTER JOIN

INNER JOIN과 OUTER JOIN의 차이는 면접에서도 자주 나옵니다. 문법만 외우지 말고 어떤 데이터가 결과에 포함되고 빠지는지 예시로 설명할 수 있어야 합니다.

### Data Modeling

- ERD
- 1:1
- 1:N
- N:M
- Normalization
- Primary Key
- Foreign Key
- Index

게시판에서는 사용자, 게시글, 댓글, 태그, 게시글-태그 관계를 설계해야 합니다.

특히 태그는 N:M 관계가 나오기 좋습니다. `posts`와 `tags`를 바로 연결하지 않고 `post_tags` 같은 중간 테이블을 두는 이유를 설명할 수 있어야 합니다.

### Transaction, Isolation, Lock

PDF에는 transaction, isolation, lock/deadlock도 포함되어 있습니다.

트랜잭션은 여러 DB 작업을 하나의 작업 단위로 묶는 개념입니다. 중간에 실패하면 전체를 되돌려야 할 때 필요합니다.

이번 프로젝트에서 트랜잭션이 필요한 예시는 다음입니다.

- 게시글 작성과 태그 연결을 함께 저장할 때
- 게시글 삭제 시 댓글/태그 관계를 함께 처리할 때
- AI 색인 요청 상태와 게시글 상태를 함께 갱신할 때
- refresh token 재발급 시 기존 토큰을 무효화하고 새 토큰을 저장할 때

Lock과 deadlock은 초급 단계에서 깊게 구현하지 않아도 되지만, 동시성 문제가 왜 생기는지는 알아두면 좋습니다.

### Scalability

- Partition
- Sharding

데이터가 많아질 때 테이블이나 DB를 나누는 방식입니다. 이번 과제에서 직접 구현할 필요는 없지만, “데이터가 커지면 어떤 문제가 생기고 어떻게 확장할 수 있는가”를 README 회고에 적을 수 있습니다.

### Caching

- Redis

Redis는 캐시로 자주 쓰입니다. 하지만 “Redis를 썼습니다”만으로는 부족합니다. 무엇을 캐싱했는지, 왜 캐싱했는지, 언제 무효화하는지 설명해야 합니다.

이번 과제에서 Redis를 적용할 수 있는 후보는 다음입니다.

- 인기 게시글 목록
- 태그별 게시글 개수
- 조회수
- refresh token
- AI 응답 캐시
- rate limit counter

## 2.4 Asynchronous Processing

PDF에는 asynchronous processing, Job Queue, RabbitMQ, Kafka가 연결되어 있습니다.

비동기 처리에는 두 층위가 있습니다.

### 언어 차원의 비동기 프로그래밍

- JavaScript/TypeScript의 async/await
- Node.js event loop
- Python FastAPI의 async def

이것은 코드 실행 방식에 가까운 개념입니다. 외부 API나 DB 응답을 기다리는 동안 다른 작업을 처리할 수 있게 합니다.

### 시스템 차원의 비동기 처리

- Job Queue
- RabbitMQ
- Kafka
- Redis Queue
- AWS SQS

이것은 오래 걸리는 작업을 큐에 넣고 나중에 처리하는 구조입니다.

이번 과제에서 큐 후보는 다음입니다.

- 게시글 작성 후 embedding 생성
- 대량 게시글 재색인
- AI Agent 실행
- 외부 URL 분석
- 알림 발송

MVP에서는 FastAPI `BackgroundTasks` 또는 단순 내부 작업으로 시작해도 됩니다. 다만 시간이 오래 걸리거나 실패해도 재시도해야 하는 작업이라면 queue 구조를 고민해야 합니다.

## 2.6 Architecture

PDF의 아키텍처 키워드입니다.

- MVC
- Layered Architecture
- Modular Monolith
- Clean Architecture
- Hexagonal Architecture
- ADR
- Microservices
- Event-driven Architecture
- CQRS

신입 단계에서는 MVC와 계층형 구조를 먼저 제대로 경험하는 것이 중요합니다.

이번 과제에서는 아래 정도를 목표로 잡을 수 있습니다.

- React: 화면과 상태 관리
- NestJS Controller: HTTP 요청/응답
- NestJS Service: 비즈니스 로직
- Repository/Prisma: DB 접근
- FastAPI Router: AI 내부 API
- FastAPI Service: LLM, RAG, MCP, Agent 로직
- PostgreSQL: 데이터 저장

더 나아가면 NestJS와 FastAPI 분리 구조 자체가 작은 서비스 분리 경험이 됩니다. 다만 이것을 무조건 microservices라고 과장하기보다, “일반 백엔드와 AI 워크로드를 분리한 구조”라고 설명하는 편이 정확합니다.

ADR은 Architecture Decision Record입니다. 왜 NestJS와 FastAPI를 분리했는지, 왜 PostgreSQL/pgvector를 선택했는지 같은 결정을 기록하는 문서입니다. README나 `/docs/adr`에 짧게 남기면 좋습니다.

이번 회의에서 선택한 구조는 TypeScript와 Python을 함께 씁니다.

- React: TypeScript 권장

따라서 팀원은 최소한 TypeScript의 타입, interface/type, async/await와 Python의 async def, Pydantic schema를 이해해야 합니다.

# 3. 풀스택 엔지니어 학습 범위 특강 정리

## 3.1 큰 범주를 먼저 잡기

풀스택 엔지니어가 알아야 할 범위는 넓습니다. 그렇다고 모든 것을 같은 깊이로 배울 수는 없습니다.

공부 순서는 다음처럼 잡는 것이 좋습니다.

1. 큰 범주를 이해한다.
2. 범주 안의 핵심 개념을 하나씩 잡는다.
3. 선택한 언어와 프레임워크에서 그 개념이 어떻게 구현되는지 확인한다.
4. 다른 기술로 넘어갈 때 같은 개념을 다시 매핑한다.

예를 들어 인증을 배웠다면, NestJS에서는 Guard와 Passport 전략으로 구현하고, FastAPI에서는 dependency와 middleware로 구현할 수 있습니다. 구현 방식은 달라도 인증의 본질은 같습니다.

## 3.2 프론트엔드 학습 범위

프론트엔드에서 볼 범주는 다음입니다.

- 프레임워크: React, Vue.js
- 렌더링: 컴포넌트 렌더링, CSR, SSR 등
- 상태 관리: local state, context, Redux, Zustand
- UI 프레임워크와 스타일링: Tailwind, MUI, shadcn, Bootstrap
- 테스트: unit, integration, E2E
- 접근성: a11y, label, keyboard, contrast

이번 프로젝트에서는 React를 선택했으므로 React 중심으로 깊게 들어가면 됩니다.

중요한 것은 지금 내가 무엇을 배우는지 분류하는 것입니다.

- JSX 문법을 배우는 것인가?
- 컴포넌트 구조를 배우는 것인가?
- 상태 관리를 배우는 것인가?
- API 호출과 로딩 상태를 배우는 것인가?
- 폼 입력과 validation을 배우는 것인가?
- 접근성과 사용자 경험을 배우는 것인가?

이렇게 분류하면 프론트엔드가 막연하게 느껴지지 않습니다.

## 3.3 언어 선택: Java, TypeScript, Python

백엔드 언어는 Java, TypeScript, Python을 생각할 수 있습니다.

### Java

Java와 Spring Boot는 한국 웹 백엔드 업계에서 매우 중요합니다. 특히 대기업, 금융권, 공공기관, 내부 업무 시스템에서는 Java/Spring 기반 시스템이 많습니다.

하지만 Java는 짧은 기간에 쉽게 익히기 어렵습니다. 문법뿐 아니라 객체지향, 패키지 구조, 클래스 설계, 예외 처리, 테스트, Spring Boot까지 연결해서 배워야 합니다.

이번 짧은 과제에서 팀원 다수가 Java/Spring을 모른다면 선택 부담이 큽니다.

### TypeScript

TypeScript는 프론트엔드와 백엔드에서 모두 사용할 수 있습니다. React와 NestJS를 함께 쓰면 같은 언어권 안에서 개발할 수 있어서 팀 커뮤니케이션 비용이 줄어듭니다.

NestJS는 Spring Boot의 구조를 많이 참고했기 때문에 나중에 Spring을 배우는 징검다리 역할도 할 수 있습니다.

### Python

Python은 AI, 데이터 처리, 프로토타이핑에 강합니다. FastAPI를 사용하면 AI 서버를 빠르게 만들 수 있고, LLM, embedding, RAG, Agent 관련 자료도 찾기 쉽습니다.

이번 회의에서 AI 기능을 FastAPI로 분리하기로 한 이유와도 연결됩니다.

## 3.4 백엔드 프레임워크

### FastAPI

Python 기반 API 프레임워크입니다.

장점:

- AI/데이터 생태계와 잘 맞음
- async/await 지원
- Pydantic 기반 request/response schema
- 자동 API 문서화
- LLM 연동과 실험이 쉬움

주의:

- 일반 게시판 전체를 FastAPI로 구현할 수도 있지만, 이번 팀 결정에서는 AI 전용 서버로 역할을 좁히는 것이 좋습니다.

### NestJS

TypeScript 기반 백엔드 프레임워크입니다.

장점:

- Module, Controller, Service 구조가 명확함
- Spring Boot와 비슷한 계층형 구조를 경험할 수 있음
- TypeScript로 프론트엔드와 백엔드를 연결하기 좋음
- 인증, CRUD, DB 연동 자료가 많음
- 프로젝트 규모에 비해 과도하게 무겁지 않음

주의:

- 객체지향적 구조와 dependency injection 개념이 낯설 수 있음
- Module 경계와 provider 주입을 이해해야 함

### Spring Boot

Java 기반 강력한 백엔드 프레임워크입니다.

장점:

- 한국 백엔드 시장에서 매우 넓게 쓰임
- 보안, 인증, 대규모 운영에 강함
- 면접과 실무 관점에서 장기적으로 중요함

주의:

- 짧은 프로젝트에서 처음 배우기에는 러닝커브가 큼
- Java와 Spring을 동시에 익혀야 함

### Next.js

React 기반 프레임워크입니다. NestJS와 이름이 비슷하지만 완전히 다릅니다.

Next.js는 프론트엔드 중심이며 서버 기능도 일부 포함합니다. NestJS는 백엔드 애플리케이션 프레임워크입니다.

## 3.5 API 설계

백엔드를 만든다는 것은 API를 설계한다는 뜻입니다.

REST API를 공부할 때는 다음을 봐야 합니다.

- 리소스 설계
- URL 구조
- HTTP method
- HTTP status code
- request/response body
- error response
- 인증이 필요한 API와 공개 API 분리

이번 프로젝트에서는 다음 두 종류의 API를 구분해야 합니다.

## 3.6 인증: JWT와 OAuth2

인증은 사용자가 누구인지 확인하는 과정입니다. 인가는 그 사용자가 어떤 작업을 할 수 있는지 판단하는 과정입니다.

주요 방식:

- Session
- JWT
- OAuth2
- OIDC

이번 과제에서는 JWT 기반 인증을 우선 구현합니다.

JWT 흐름:

1. 사용자가 로그인한다.
2. NestJS가 이메일/비밀번호를 검증한다.
3. NestJS가 access token과 refresh token을 발급한다.
4. React는 access token을 API 요청 header에 넣는다.
5. NestJS Guard가 token을 검증한다.
6. 검증된 사용자만 게시글 작성, 수정, 삭제를 할 수 있다.

FastAPI는 사용자 로그인을 직접 처리하지 않습니다. FastAPI는 NestJS가 검증한 user context를 받아 AI 작업을 수행합니다.

## 3.7 실시간 기능: WebSocket과 SSE

실시간 기능이 필요한 경우 WebSocket과 SSE를 검토합니다.

- 양방향 통신

이번 AI 게시판에서는 필수는 아닙니다. 하지만 AI Agent 실행 시간이 길어지면 “진행 중” 상태를 보여주기 위해 SSE를 사용할 수 있습니다.

# 4. 나만무 프로젝트 주제 선정 특강 정리

## 4.1 주제 고민에 시간을 너무 많이 쓰지 말 것

## 4.2 기술 중심의 주제가 좋다

사업성이 있어 보이는 아이디어도 좋지만, 개발자 포트폴리오에서는 기술적으로 설명할 수 있어야 합니다.

## 4.3 가장 많이 지적받는 문제: 스코프가 너무 작다

5명이 5주 동안 하는 프로젝트는 한 명이 며칠 만에 만들 수 있는 서비스와 달라야 합니다.

단순히 기능 개수를 늘리는 것은 좋은 방향이 아닙니다.

나쁜 스코프 확장:

- 게시판을 여러 개 만든다.
- 메뉴를 많이 붙인다.
- CRUD 화면을 많이 만든다.
- AI가 하루 만에 만들어줄 수준의 기능만 나열한다.

좋은 스코프 확장:

- 트래픽을 실어보고 병목을 분석한다.
- 데이터 처리량을 늘리고 성능 개선을 한다.
- 캐시, 큐, 비동기 처리, 검색, 추천, 실시간 기능을 설계한다.
- 단순 기능 구현을 넘어 운영 가능한 서비스 구조를 고민한다.

## 4.4 발표와 데모 가능성

프로젝트는 발표장에서 보여줄 수 있어야 합니다.

위험한 주제:

- 특정 장소나 날씨에 의존하는 서비스
- 센서나 하드웨어가 필요한 서비스
- GPS가 안정적으로 잡혀야 하는 서비스
- 네트워크 품질에 지나치게 의존하는 실시간 서비스
- 발표장에서 데이터 준비가 어려운 서비스

데모 가능성 체크:

- 발표장 안에서 핵심 기능을 보여줄 수 있는가?
- 외부 환경에 지나치게 의존하지 않는가?
- 네트워크가 불안정해도 대체 시나리오가 있는가?
- 시연 데이터와 시연 계정을 준비할 수 있는가?
- 핵심 가치가 3-5분 안에 전달되는가?

이번 AI 게시판은 데모하기 좋은 편입니다. 단, OpenAI API나 외부 API가 실패할 때를 대비해 fallback 데이터나 녹화/캡처를 준비해야 합니다.

# 5. 우리 AI 게시판 과제에 적용하기

## 5.1 현재 회의 결정과 특강 내용의 연결

회의 결정:

- Frontend: React
- Backend: NestJS
- AI Server: FastAPI
- Database: PostgreSQL + pgvector
- Authentication: JWT
- LLM: GPT-4.1 mini 이상
- Infra: Docker + AWS

특강 기준으로 보면 이 결정은 합리적입니다.

- Java/Spring은 취업 시장에서 강하지만, 짧은 기간과 팀 숙련도를 고려하면 부담이 큽니다.
- NestJS는 TypeScript 기반이고 Spring 구조를 참고했기 때문에 백엔드 구조 학습에 좋습니다.
- FastAPI는 AI 기능, LLM 호출, embedding, MCP, Agent 구현에 적합합니다.
- PostgreSQL/pgvector는 RDBMS 학습과 RAG 기능을 동시에 만족합니다.
- Docker/AWS는 배포와 운영 관점까지 경험할 수 있게 합니다.

---

## 5.2 추천 아키텍처

```text
React
  |
  | public REST API
  v
NestJS
  |  - JWT 인증
  |  - 회원/게시글/댓글/태그
  |  - 일반 검색/페이징
  |  - FastAPI 내부 호출
  v
PostgreSQL + pgvector

NestJS
  |
  | internal API + service token
  v
FastAPI
  |  - GPT 호출
  |  - embedding 생성
  |  - RAG 검색
  |  - MCP tool
  |  - Agent workflow
  v
OpenAI API / External APIs
```

핵심 원칙:

- React는 NestJS만 호출한다.
- FastAPI는 외부 공개 API가 아니라 내부 AI 서버로 둔다.
- NestJS는 인증과 게시판 도메인의 주인이다.
- FastAPI는 AI 실행과 AI 로그의 주인이다.
- PostgreSQL은 하나를 쓰되 일반 테이블과 AI 테이블의 책임을 나눈다.

# 9. 최종 정리

코치님 특강의 핵심은 “많은 기술을 얕게 나열하지 말고, 선택한 기술 안에서 핵심 개념을 실제로 경험하고 설명할 수 있게 하라”는 것입니다.

이번 팀의 방향인 React + NestJS + FastAPI + PostgreSQL/pgvector는 짧은 기간 안에 AI 게시판을 완성하면서도 풀스택 개념을 경험하기 좋은 구조입니다.

다만 이 구조가 설득력을 가지려면 다음이 반드시 필요합니다.

- NestJS와 FastAPI의 역할 분리
- JWT 인증 흐름
- PostgreSQL 데이터 모델링
- pgvector를 이용한 RAG 검색
- MCP 외부 도구 1개 이상
- Agent 반복 제한과 실행 로그
- Docker/AWS 배포 계획
- 사용자와 데모 시나리오

결국 목표는 “AI를 붙인 게시판”이 아니라, **기술적으로 설명할 수 있고 실제로 동작하며 사용자와 운영까지 고려한 프로젝트**를 만드는 것입니다.
