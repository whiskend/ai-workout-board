# 06. Test Guide

이 문서는 테스터 역할을 수행할 때 사용하는 문서다.

테스트 단계의 목표는 “코드가 빌드된다”가 아니라, **사용자 관점에서 실제로 동작한다**를 확인하는 것이다.

---

## 1. 테스트의 세 종류

### 1.1 빌드/타입 테스트

코드가 깨지지 않았는지 확인한다.

```bash
npm run build
npm run typecheck
npm run lint
```

프로젝트에 맞는 명령어를 사용한다.

### 1.2 수동 기능 테스트

브라우저나 API 클라이언트로 실제 흐름을 확인한다.

```text
로그인
-> 기능 실행
-> 결과 확인
```

### 1.3 실패 케이스 테스트

일부러 실패 상황을 만든다.

```text
로그인 안 함
권한 없는 사용자
잘못된 입력
외부 서버 꺼짐
데이터 없음
```

## 3. 테스터가 봐야 할 것

```text
기능이 성공할 때:
- 화면에 원하는 결과가 나오는가?
- DB에 저장되어야 하는 데이터가 저장되는가?
- 저장하지 말아야 하는 데이터는 저장되지 않는가?

기능이 실패할 때:
- 화면이 깨지지 않는가?
- 적절한 에러 메시지가 나오는가?
- 서버가 죽지 않는가?
- 보안상 민감한 정보가 노출되지 않는가?

기존 기능:
- 새 기능 때문에 기존 기능이 깨지지 않았는가?
```

## 4. 테스트 단계 Codex 프롬프트

```text
AGENTS.md와 docs/06-test-guide.md를 읽어줘.
지금은 테스트 단계야.
아직 코드는 수정하지 마.

이번 변경사항을 내가 직접 확인할 수 있게 테스트 시나리오를 만들어줘.

아래 형식으로 답해줘.

역할 확인:
- 현재 단계:
- 사용자의 역할:
- Codex의 역할:
- 지금 사용자가 해야 할 일:
- 아직 하면 안 되는 일:

테스트 계획:
1. 실행해야 할 빌드/테스트 명령어
2. 성공 케이스
3. 실패 케이스
4. 엣지 케이스
5. 실제 결과를 기록하는 방법
6. 통과하면 커밋해도 되는 기준
```

```text
테스트 중 에러가 났어.
아직 코드는 수정하지 말고 원인 후보부터 설명해줘.

내가 한 행동:

에러 메시지:

예상 결과:

실제 결과:

관련 파일 또는 화면:
```

---

## 6. 완료 기준

테스트 완료는 아래가 모두 만족될 때다.

```text
- 빌드/타입 체크가 통과했다.
- 핵심 성공 케이스가 통과했다.
- 최소 1개 이상의 실패 케이스를 확인했다.
- 기존 핵심 기능이 깨지지 않았다.
- 사용자가 결과를 직접 확인했다.
```
