## 2.3 AI 기능 최소 기준

AI 기능은 너무 크게 잡지 않는다.

### RAG 최소 기준

```text
이전 운동 기록을 DB에서 검색한다.
        ↓
검색된 기록을 AI prompt에 넣는다.
        ↓
AI가 현재 기록과 이전 기록을 비교한다.
```

pgvector가 늦어지면 먼저 구조화 DB 검색으로 대체한다.

```text
같은 사용자
+ 같은 운동명
+ 최근 날짜
```

이 방식도 발표에서는 “RAG의 검색 단계 최소 구현”으로 설명할 수 있다.  
pgvector는 되면 붙이고, 막히면 개선 아이디어로 남긴다.

### MCP 최소 기준

MCP는 외부 tool 1개만 만든다.

후보:

- 운동명 정규화 tool
- 외부 운동 정보 조회 tool
- 날씨/컨디션 참고 tool

가장 단순한 선택은 **운동명 정규화 tool**이다.

```text
"벤치", "벤치프레스", "bench press"
        ↓
"벤치프레스"로 정규화
```

### Agent 최소 기준

완전 자율 Agent를 만들지 않는다.

정해진 workflow로 만든다.

```text
1. 현재 게시글 조회
2. 이전 기록 검색
3. MCP tool 호출
4. AI 분석 요청
5. 다음 목표 추천
6. 분석 결과 저장
```

Agent라고 설명할 수 있는 포인트는 이것이다.

```text
한 번의 요청 안에서 여러 단계를 순서대로 실행하고,
중간 결과를 상태로 들고,
마지막 추천 결과를 만든다.
```

## 6. 컷라인

시간이 부족하면 아래 순서로 자른다.

## 6.1 자르면 안 되는 것

- 로그인
- 게시글 작성/조회
- AI 분석 결과
- RAG/MCP/Agent 설명 가능한 최소 구현
- README
- 발표자료

## 6.2 줄일 수 있는 것

- 댓글은 작성만 구현하고 수정은 생략
- 태그는 문자열 배열 수준으로 단순화
- 검색은 제목/운동명만 우선
- 페이징은 limit/offset만 우선
- MCP는 tool 1개만
- Agent는 정해진 workflow만

## 6.3 과감히 버릴 수 있는 것

- AWS 완성 배포
- UI 고도화
- 그래프
- 알림
- 소셜 로그인
- 관리자 기능

cd backend

npm run test -- --runInBand

cd ../frontend
npm run build

## 8. 최종 데모 목표

최종적으로 보여줄 흐름은 하나다.

```text
회원가입
-> 로그인
-> 운동 기록 작성
-> 게시글 목록 확인
-> 게시글 상세 확인
-> AI 분석 실행
-> 이전 기록과 비교
-> 다음 운동 목표 추천
-> 댓글/태그/검색 확인
```

이 흐름이 안정적으로 동작하면 성공이다.
