# 00. How to Use

이 문서는 역할 기반 Codex 하네스를 사용하는 방법을 설명한다.

목표는 Codex에게 모든 판단을 맡기는 것이 아니라, 사용자가 **기획자 / 설계자 / 리뷰어 / 테스터** 역할을 배우면서 프로젝트를 진행하는 것이다.

---

## 1. 처음 세팅

프로젝트 루트에 아래 파일을 둔다.

```text
AGENTS.md
docs/
```

그다음 `docs/01-project-context.md`를 현재 프로젝트에 맞게 채운다.

최소한 아래는 작성한다.

```text
- 프로젝트 목적
- 기술 스택
- 현재 구현된 기능
- 주요 폴더 구조
- 주요 API
- 주요 DB 모델
- 실행 / 테스트 명령어
- 건드리면 위험한 파일
```

---

## 2. 매 작업의 기본 흐름

모든 작업은 아래 흐름을 따른다.

```text
1. 역할 확인
2. 기획 또는 설계 설명
3. 사용자가 결정
4. Codex가 구현
5. Codex가 변경사항 설명
6. 사용자가 리뷰
7. 사용자가 테스트
8. 문서/커밋 정리
```

---

## 3. Codex에게 매번 먼저 시킬 말

새 작업을 시작할 때는 바로 구현시키지 말고 아래처럼 말한다.

```text
AGENTS.md와 관련 docs를 먼저 읽어줘.
아직 코드는 수정하지 마.

먼저 내가 지금 어떤 역할을 해야 하는지 알려주고,
이 작업을 진행하기 전에 내가 결정해야 할 것을 정리해줘.
```

---

## 4. 사용자가 직접 결정해야 하는 것

사용자는 아래 결정을 Codex에게 넘기지 않는다.

```text
- 이 기능을 지금 할지 나중에 할지
- 어떤 설계안으로 갈지
- DB를 바꿀지 말지
- 새 API를 만들지 말지
- 기존 구조를 유지할지 바꿀지
- 이번 작업 범위를 어디까지로 자를지
```

Codex는 선택지를 제시할 수 있지만, 최종 결정은 사용자가 한다.

---

## 5. 기능 하나를 시작할 때 절차

```text
1. 기능 이름을 정한다.
2. 왜 필요한지 한 문장으로 말한다.
3. Codex에게 기획 질문을 뽑게 한다.
4. Codex에게 설계안 2~3개를 받는다.
5. 사용자가 하나를 고른다.
6. Codex가 수정할 파일과 테스트 방법을 보고한다.
7. 사용자가 승인한다.
8. Codex가 구현한다.
```

---

## 6. 기능 하나를 끝냈을 때 절차

```text
1. Codex에게 변경 파일별 요약을 받는다.
2. Codex에게 위험한 부분을 찾게 한다.
3. 사용자가 핵심 diff를 확인한다.
4. 빌드/테스트를 실행한다.
5. 실제 화면 또는 API로 수동 테스트한다.
6. 문서를 최신화한다.
7. 커밋한다.
```

---

## 7. 길을 잃었을 때 쓰는 프롬프트

```text
지금 내가 뭘 하고 있는지 헷갈려.
아직 코드는 수정하지 마.

현재 상황을 기준으로 아래만 쉽게 정리해줘.

1. 지금 프로젝트는 어느 단계인지
2. 내가 지금 맡아야 할 역할은 무엇인지
3. 이 역할에서 내가 해야 하는 일 3개
4. 아직 하면 안 되는 일 3개
5. 다음에 Codex에게 시킬 가장 작은 작업 1개
```

---

## 8. 핵심 원칙

사용자는 Codex 답변을 그대로 믿고 넘기는 사람이 아니다.

사용자는 Codex의 도움을 받아 다음을 판단하는 사람이다.

```text
이 기능이 필요한가?
이 구조가 맞는가?
이 변경이 위험하지 않은가?
실제로 동작하는가?
발표나 문서로 설명 가능한가?
```
