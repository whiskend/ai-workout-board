# 00. 이 하네스 사용법

이 문서는 초보 개발자가 Codex와 함께 **설계를 배우면서 프로젝트를 확장하는 방법**을 설명한다.

이 하네스의 목적은 Codex에게 모든 결정을 맡기는 것이 아니다.

```text
Codex가 구현한다.
하지만 사용자는 기획자, 설계자, 리뷰어, 테스터 역할을 배운다.
Codex는 그 역할을 할 수 있도록 질문과 기준을 제공한다.
```

---

## 1. 하네스란 무엇인가?

여기서 하네스는 프로젝트를 안전하게 확장하기 위한 문서 세트다.

Codex는 프로젝트가 커질수록 전체 맥락을 놓칠 수 있다. 그래서 다음을 문서로 고정한다.

- 이 프로젝트가 무엇을 만드는지
- 현재 어디까지 구현됐는지
- 어떤 구조를 지켜야 하는지
- API와 DB가 어떻게 생겼는지
- AI/RAG/MCP/Agent를 어디까지 구현했다고 말할 수 있는지
- Codex가 작업 전에 무엇을 보고해야 하는지
- 사용자가 무엇을 검수해야 하는지

---

## 2. 문서 읽는 순서

작업을 시작할 때는 아래 순서로 읽는다.

```text
AGENTS.md
-> docs/01-current-state.md
-> 작업과 관련된 설계 문서
-> docs/08-codex-workflow.md
-> docs/09-review-test-checklist.md
```

예를 들어 RAG 최소 구현을 한다면 다음 문서를 읽는다.

```text
AGENTS.md

-> docs/03-domain-design.md
-> docs/04-architecture.md
-> docs/07-ai-rag-design.md
-> docs/08-codex-workflow.md
-> docs/09-review-test-checklist.md
```

README를 고칠 때는 다음 문서를 읽는다.

```text
AGENTS.md
-> docs/01-current-state.md
-> docs/02-product-planning.md
-> docs/04-architecture.md
-> docs/07-ai-rag-design.md
-> README.md
```

## 3. 기능 하나를 시작하는 절차

기능을 시작할 때는 Codex에게 바로 “구현해줘”라고 하지 않는다.

먼저 아래 순서로 진행한다.

### 1단계: 기능 이름 정하기

예시:

```text
RAG 최소 구현을 정리하고 싶다.
```

좋은 기능 이름은 작고 명확하다.

좋은 예:

```text
- 같은 운동명의 이전 기록 조회 로직 정리
- OpenAI API 연결
- 운동명 정규화 함수 추가
- AI 분석 실패 시 프론트 화면 처리
```

나쁜 예:

```text
- AI 기능 완성해줘
- RAG/MCP/Agent 다 구현해줘
- 프로젝트 구조 좋게 바꿔줘
```

---

### 2단계: Codex에게 설계 질문 시키기

프롬프트 예시:

```text
AGENTS.md와 docs를 읽고, 아직 구현하지 말고 먼저 설계 질문만 해줘.
이번 작업은 RAG 최소 구현 정리야.
내가 결정해야 할 것, 선택지, 장단점, 추천안을 알려줘.
```

Codex는 다음을 말해야 한다.

- 이 기능이 왜 필요한지
- 지금 구조에서 어디에 들어가는지
- 수정할 파일이 무엇인지
- DB 변경이 필요한지
- API 변경이 필요한지
- 구현 난이도는 어떤지
- 초보자가 이해해야 할 핵심은 무엇인지

---

### 3단계: 작은 결정 내리기

사용자는 Codex가 준 선택지를 보고 결정한다.

예시:

```text
이번에는 pgvector 없이 구조화 검색 기반으로만 가자.
운동명 정규화는 함수 수준으로만 만들고, MCP 서버는 발표용 설계로 남기자.
```

이 결정이 중요하다. 이렇게 해야 Codex가 범위를 넘지 않는다.

---

### 4단계: 구현 전 보고 받기

Codex는 반드시 아래를 먼저 말해야 한다.

```text
이번 작업 목표:
수정할 파일:
새로 만들 파일:
변경하지 않을 영역:
영향 받는 API:
영향 받는 DB:
테스트 방법:
설계상 주의점:
초보자가 이해해야 할 핵심:
```

---

### 5단계: 승인 후 구현하기

사용자가 “진행해줘”라고 말한 뒤 구현한다.

구현 중 Codex가 새로운 의존성 설치, DB schema 변경, API 응답 변경 같은 큰 결정을 해야 하면 멈추고 다시 확인해야 한다.

---

## 4. 기능 하나를 끝내는 절차

기능이 끝났다고 바로 다음 기능으로 넘어가지 않는다.

아래 순서로 마무리한다.

### 1단계: 구현 요약 받기

Codex에게 다음을 정리하게 한다.

```text
완료한 작업:
수정한 파일:
새로 만든 파일:
변경하지 않은 영역:
설계 이유:
코드 흐름:
확인 방법:
주의할 점:
```

---

### 2단계: 직접 검수하기

`docs/09-review-test-checklist.md`를 보고 확인한다.

예시 질문:

```text
Controller가 Prisma를 직접 호출하지 않았나?
React가 FastAPI를 직접 호출하지 않았나?
AI 분석 실패 시 화면이 깨지지 않나?
기존 게시글 작성/상세 기능이 깨지지 않았나?
```

---

### 3단계: 빌드와 통합 흐름 확인하기

최소 확인:

```bash
cd backend
npm run build

cd ../frontend
npm run build
```

AI 서버 문법 확인:

```bash
cd ai-server
python -m py_compile app/main.py app/routers/analysis.py app/schemas/analysis.py app/services/analysis_service.py
```

통합 흐름:

```text
회원가입
-> 로그인
-> 게시글 작성
-> 상세 화면
-> AI 분석 버튼 클릭
-> 결과 표시
```

---

### 4단계: 문서 업데이트하기

기능을 구현했으면 문서도 같이 업데이트한다.

예시:

- RAG 흐름을 정리했다면 `docs/07-ai-rag-design.md`와 `README.md` 업데이트
- API가 추가됐다면 `docs/05-api-design.md`와 `README.md` 업데이트
- DB가 바뀌었다면 `docs/06-database-design.md` 업데이트
- 프로젝트 상태가 바뀌었다면 `docs/01-current-state.md` 업데이트

---

## 5. 사용자가 직접 결정해야 하는 것

초보자라도 다음 결정은 직접 해야 한다.

- 이번 기능이 발표에 꼭 필요한가?
- 지금 구현할 것인가, 나중 확장으로 둘 것인가?
- DB schema를 바꿀 것인가?
- API 응답 형식을 바꿀 것인가?
- README에서 어디까지 “구현”이라고 말할 것인가?
- RAG/MCP/Agent를 과장하지 않고 어떻게 표현할 것인가?

Codex는 결정할 수 있게 도와줄 뿐이다.

---

## 6. Codex에게 자주 시킬 질문

기능 시작 전 질문:

```text
이 기능은 지금 MVP에 꼭 필요한가?
```

```text
현재 구조에서 이 기능은 어느 도메인에 들어가야 해?
```

```text
DB 변경 없이 구현할 수 있어?
```

```text
API 응답 형식을 바꾸지 않고 구현할 수 있어?
```

```text
초보자가 이해해야 할 코드 흐름을 먼저 설명해줘.
```

구현 후 질문:

```text
내가 리뷰해야 할 체크리스트를 만들어줘.
```

```text
성공 케이스와 실패 케이스를 나눠서 테스트 방법을 알려줘.
```

```text
README 발표용으로 이 기능을 어떻게 설명하면 과장되지 않아?
```

## 7. 추천 진행 순서

현재 상태에서 추천 진행 순서는 다음과 같다.

```text
1. 문서 하네스 적용
2. README 발표용 구조 정리
3. RAG 최소 구현 정리
4. OpenAI API 연결
5. 운동명 정규화 tool 구현
6. Agent workflow 문서/코드 흐름 정리
7. 데모 안정화
8. 발표자료/스크린샷 정리
```

시간이 부족하면 1~3과 README만 해도 발표 구조는 잡힌다.

---

## 8. 첫 작업 추천 프롬프트

```text
AGENTS.md와 docs 문서를 먼저 읽어줘.
아직 코드를 수정하지 말고, 현재 프로젝트 상태를 기준으로 RAG 최소 구현을 어떻게 정리하면 좋을지 설계 코치 역할로 설명해줘.

목표는 pgvector를 당장 붙이는 것이 아니라,
로그인한 사용자 + 같은 운동명 + 최근 기록 N개를 AI 분석 재료로 사용하는 구조화 검색 기반 RAG 흐름을 발표 가능하게 정리하는 거야.

다음 형식으로 먼저 답해줘.

이번 작업 목표:
수정할 파일:
새로 만들 파일:
변경하지 않을 영역:
영향 받는 API:
영향 받는 DB:
테스트 방법:
설계상 주의점:
초보자가 이해해야 할 핵심:
내가 결정해야 할 것:
```
