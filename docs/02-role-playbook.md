# 02. Role Playbook

이 문서는 사용자가 수행할 역할과 Codex가 도와주는 방식을 정리한다.

---

## 1. 전체 역할 흐름

```text
기획자
  ↓
설계자
  ↓
구현 승인자
  ↓
리뷰어
  ↓
테스터
  ↓
문서 정리자
```

사용자의 핵심 역할은 Codex가 만든 결과를 그대로 받아들이는 것이 아니라, 각 단계에서 필요한 판단을 하는 것이다.

---

## 2. 기획자 역할

기획자는 “무엇을 만들지”와 “무엇을 안 만들지”를 결정한다.

### 사용자가 해야 할 일

```text
- 이 기능이 왜 필요한지 말하기
- MVP에 포함할지 판단하기
- 지금 할 기능과 나중에 할 기능 나누기
- 성공 기준 정하기
```

### Codex가 도와줄 일

```text
- 기능의 목적 정리
- 사용자 시나리오 작성
- 필수 / 선택 / 제외 기능 분류
- 발표용 설명 문장 작성
```

### Codex에게 시킬 말

```text
아직 코드 수정하지 마.
지금은 기획 단계야.
이 기능이 MVP에 필요한지 판단할 수 있게 질문과 선택지를 정리해줘.
```

---

## 3. 설계자 역할

설계자는 “어떤 구조로 만들지”를 결정한다.

### 사용자가 해야 할 일

```text
- 어떤 설계안으로 갈지 결정하기
- DB를 바꿀지 말지 결정하기
- 새 API를 만들지 말지 결정하기
- 기존 구조를 유지할지 바꿀지 결정하기
```

### Codex가 도와줄 일

```text
- 설계안 2~3개 제시
- 장단점 비교
- 추천안 제시
- 영향 받는 파일/API/DB 정리
- 변경하지 말아야 할 영역 정리
```

### Codex에게 시킬 말

```text
아직 코드 수정하지 마.
지금은 설계 단계야.
이 기능을 구현하는 설계안 2~3개와 장단점을 정리해줘.
내가 결정해야 할 것을 따로 표시해줘.
```

---

## 4. 구현 승인자 역할

구현 승인자는 Codex가 실제 코드를 수정해도 되는지 허락한다.

### 사용자가 해야 할 일

```text
- 수정할 파일 목록 확인
- DB/API 영향 확인
- 작업 범위가 너무 크지 않은지 확인
- “진행해” 또는 “아직 하지 마”라고 명확히 말하기
```

### Codex가 도와줄 일

```text
- 구현 전 보고
- 수정 범위 제한
- 작업 후 변경사항 설명
```

### 승인 예시

```text
좋아. A안으로 진행해.
단, 이번에는 DB schema는 바꾸지 말고, 기존 API 흐름도 유지해.
```

---

## 5. 리뷰어 역할

리뷰어는 Codex가 만든 코드가 설계와 맞는지 확인한다.

### 사용자가 해야 할 일

```text
- 변경된 파일 확인
- 설계 결정이 지켜졌는지 확인
- 권한/인증/데이터 흐름 문제 확인
- 당장 고쳐야 할 것과 나중에 할 것 구분
```

### Codex가 도와줄 일

```text
- 변경사항 요약
- 위험한 부분 찾기
- 리뷰 체크리스트 제공
- 수정 필요 여부 분류
```

### Codex에게 시킬 말

```text
아직 코드 수정하지 마.
지금은 리뷰 단계야.
이번 변경사항이 우리가 정한 설계와 맞는지 검토해줘.
```

---

## 6. 테스터 역할

테스터는 기능이 실제로 동작하는지 확인한다.

### 사용자가 해야 할 일

```text
- 빌드/테스트 명령어 실행
- 성공 케이스 확인
- 실패 케이스 확인
- 실제 화면 또는 API로 흐름 확인
```

### Codex가 도와줄 일

```text
- 테스트 시나리오 작성
- 예상 결과 정리
- 에러가 났을 때 원인 후보 정리
```

### Codex에게 시킬 말

```text
지금은 테스트 단계야.
내가 직접 따라 할 수 있는 성공/실패 테스트 시나리오를 만들어줘.
```

---

## 7. Codex가 항상 말해야 하는 것

모든 답변 시작에 아래를 말한다.

```text
지금 당신의 역할은 [역할]입니다.
당신이 지금 해야 할 일은 [행동]입니다.
아직 하면 안 되는 일은 [금지 행동]입니다.
```

이 규칙이 사용자가 길을 잃지 않게 하는 핵심 장치다.
