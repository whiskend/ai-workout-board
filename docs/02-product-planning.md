# 02. 제품 기획 문서

이 문서는 **AI 운동 기록 인증 게시판**이 무엇을 해결하려는 서비스인지 정리한다.

코드를 짜기 전에 “왜 이 기능이 필요한지”를 먼저 생각하기 위한 문서다.

---

## 1. 서비스 한 줄 소개

```text
운동 기록을 게시글처럼 남기고, 이전 기록을 바탕으로 다음 운동 목표를 추천받는 AI 운동 기록 게시판
```

---

## 2. 해결하려는 문제

운동 초보자는 꾸준히 운동을 해도 다음을 판단하기 어렵다.

- 지난번보다 내가 성장했는지
- 무게를 올려도 되는지
- 같은 무게로 반복 수를 더 채워야 하는지
- 다음 운동 목표를 어떻게 잡아야 하는지
- 이전 기록을 어떻게 비교해야 하는지

이 프로젝트는 이 문제를 “기록”과 “비교”로 해결한다.

사용자가 운동 기록을 남기면, 서비스가 이전 기록을 찾아 비교하고 다음 목표를 제안한다.

---

## 3. 핵심 주제

핵심 주제는 다음이다.

```text
운동 기록 기반 점진적 과부하 보조
```

점진적 과부하는 간단히 말하면 다음과 같다.

```text
이전보다 조금씩 더 많이 하거나,
더 무겁게 하거나,
더 안정적으로 반복해서 성장하는 방식
```

이 프로젝트는 전문적인 운동 처방 서비스가 아니다.

사용자가 남긴 기록을 기반으로 다음 운동에서 시도할 목표를 정리해주는 보조 서비스다.

---

## 4. 핵심 사용자

### 1순위 사용자

```text
운동을 시작한 지 오래되지 않았고,
자신의 기록을 꾸준히 남기며 성장감을 느끼고 싶은 사람
```

이 사용자는 다음에 어려움을 느낀다.

- 기록을 남기긴 하는데 비교가 어렵다.
- 지난번보다 발전했는지 잘 모르겠다.
- 다음 운동 목표를 정하기 어렵다.
- 운동 앱은 복잡하고, 단순한 게시판처럼 기록하고 싶다.

---

### 2순위 사용자

```text
운동 기록을 포트폴리오처럼 정리하고 싶은 학습자 또는 개발 프로젝트 발표자
```

이 프로젝트는 학습용 프로젝트이므로, 사용자 기능뿐 아니라 개발 구조도 중요하다.

발표에서 보여줄 포인트:

- 인증
- 게시글 CRUD
- DB 관계 설계
- AI 서버 분리
- RAG 흐름
- MCP/tool 후보
- Agent workflow

---

## 5. MVP 기능

MVP는 “최소한 이 정도는 있어야 서비스라고 부를 수 있는 기능”이다.

현재 MVP 기준은 다음과 같다.

| 기능 | 설명 | 현재 상태 |
|---|---|---|
| 회원가입 | 사용자 계정 생성 | 완료 |
| 로그인 | JWT 발급 | 완료 |
| 내 정보 조회 | 로그인 사용자 확인 | 완료 |
| 운동 기록 작성 | 게시글 형태로 운동 기록 저장 | 완료 |
| 운동 기록 목록 | 작성된 운동 기록 확인 | 완료 |
| 운동 기록 상세 | 운동 세트 기록 확인 | 완료 |
| 운동 기록 수정/삭제 | 작성자만 수정/삭제 | 완료 |
| 제목/운동명 검색 | 기록 검색 | 완료 |
| AI 분석 요청 | 현재 기록과 이전 기록 비교 요청 | demo 완료 |
| AI 분석 결과 표시 | summary, recommendation, nextGoal 표시 | 완료 |

---

## 6. 나중에 붙일 기능

현재 바로 구현하지 않아도 되는 기능이다.

| 기능 | 설명 | 우선순위 |
|---|---|---|
| OpenAI API 연결 | demo 분석 대신 실제 LLM 응답 생성 | 높음 |
| 구조화 검색 기반 RAG 정리 | 같은 사용자 + 같은 운동명 + 최근 기록 N개 활용 | 높음 |
| 운동명 정규화 tool | 벤치/bench press/벤치프레스 통일 | 중간 |
| Agent workflow 정리 | 분석 과정을 단계별 workflow로 설명 | 중간 |
| pgvector | embedding 기반 유사도 검색 | 낮음, 확장 기능 |
| 이미지 인증 | 운동 인증 사진 업로드 | 낮음 |
| 좋아요/댓글 | 커뮤니티 기능 강화 | 낮음 |
| 루틴 관리 | 운동 루틴 템플릿 저장 | 낮음 |
| 통계 그래프 | 운동별 성장 그래프 | 낮음 |

---

## 7. 이번 프로젝트에서 하지 않을 기능

범위를 줄이기 위해 다음 기능은 하지 않는다.

- 부상 진단
- 질병/의학 조언
- 전문 트레이너 수준 운동 처방
- 식단 처방
- 유료 결제
- SNS 팔로우 시스템
- 실시간 채팅
- 복잡한 추천 알고리즘
- 완성형 벡터 DB 기반 RAG 시스템

이 프로젝트의 목적은 “운동 초보자를 위한 기록 기반 성장 보조”다.

---

## 8. 의학 조언이 아니라는 경계

이 서비스는 다음을 하지 않는다.

```text
- 통증의 원인 진단
- 부상 여부 판단
- 병원 진료 대체
- 사용자의 신체 상태에 대한 의학적 결론
```

AI 응답도 다음처럼 안전하게 제한한다.

좋은 응답:

```text
지난 기록보다 반복 수가 늘었습니다. 다음 운동에서는 같은 무게로 목표 반복 수를 안정적으로 채우는 것을 우선해보세요.
```

피해야 할 응답:

```text
어깨 통증은 특정 질환 때문입니다. 이 운동을 계속해도 됩니다.
```

README와 발표에서도 다음 문장을 넣는 것이 좋다.

```text
이 프로젝트는 의학 조언이나 부상 진단 서비스가 아니라, 사용자의 운동 기록을 기반으로 다음 목표를 정리해주는 학습용 AI 보조 서비스입니다.
```

---

## 9. 발표에서 강조할 포인트

발표에서는 기능 개수보다 구조를 강조한다.

강조할 내용:

1. React, NestJS, FastAPI, PostgreSQL을 역할별로 분리했다.
2. React는 FastAPI를 직접 호출하지 않고, NestJS가 인증과 데이터 조회를 담당한다.
3. AI 분석은 현재 기록과 이전 기록을 함께 전달받는다.
4. 같은 사용자와 같은 운동명의 최근 기록을 분석 재료로 사용한다.
5. 현재는 demo 분석이지만 OpenAI API로 확장 가능한 구조다.
6. RAG/MCP/Agent는 과장하지 않고 최소 구현/설계 흐름으로 설명한다.

---

## 10. 기능을 추가하기 전 기획 질문

새 기능을 추가하기 전에 Codex는 다음 질문을 던져야 한다.

```text
이 기능은 MVP에 필요한가?
발표에 도움이 되는가?
지금 구현해야 하는가, 나중 확장으로 둬도 되는가?
기존 기능을 깨뜨릴 위험이 있는가?
DB 변경이 필요한가?
사용자가 이 기능을 실제로 어떻게 사용하게 되는가?
```

## 11. 현재 다음 기획 결정

현재 남은 작업 중 가장 좋은 다음 결정은 다음이다.

```text
pgvector 없이, 현재 DB 구조를 이용해서 "같은 사용자 + 같은 운동명 + 최근 기록 N개"를 AI 분석 재료로 사용하는 구조화 검색 기반 RAG를 먼저 정리한다.
```

이렇게 하면 구현 범위는 작고, 발표에서는 AI 구조를 설명하기 좋아진다.
