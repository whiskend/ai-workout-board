# 03. Planning Guide

이 문서는 기획자 역할을 수행할 때 사용하는 문서다.

기획 단계의 목표는 “무엇을 만들지”와 “무엇을 만들지 않을지”를 정하는 것이다.

---

## 1. 기획 단계에서 코드 수정 금지

기획 단계에서는 코드를 수정하지 않는다.

이 단계의 산출물은 코드가 아니라 아래와 같은 판단이다.

```text
- 이 기능이 왜 필요한가?
- 누가 쓰는가?
- MVP에 꼭 필요한가?
- 성공 기준은 무엇인가?
- 이번에 하지 않을 것은 무엇인가?
```

---

## 2. 기능 기획 카드

새 기능을 시작할 때 아래를 채운다.

```md
# 기능 기획 카드

## 기능 이름


## 이 기능이 해결하는 문제


## 핵심 사용자


## 사용자가 하는 행동

1.
2.
3.

## 이 기능이 성공한 상태


## MVP 포함 여부

- [ ] 필수
- [ ] 있으면 좋음
- [ ] 지금은 제외

## 이번 작업에서 하지 않을 것

- 
- 
```

---

## 3. MVP 판단 기준

기능을 아래 셋 중 하나로 분류한다.

| 구분 | 의미 | 예시 |
|---|---|---|
| 필수 | 없으면 서비스 핵심 흐름이 안 됨 | 로그인, 핵심 데이터 생성 |
| 있으면 좋음 | 있으면 완성도가 올라감 | 필터, 정렬, 예쁜 UI |
| 지금은 제외 | 발표/마감 전에 위험함 | 대규모 리팩터링, 복잡한 추천 시스템 |

---

## 4. 기획 단계 Codex 프롬프트

```text
AGENTS.md와 docs/03-planning-guide.md를 읽어줘.
아직 코드는 수정하지 마.

지금은 기획 단계야.
이번 기능은 [기능 이름]이야.

아래 형식으로 정리해줘.

역할 확인:
- 현재 단계:
- 사용자의 역할:
- Codex의 역할:
- 지금 사용자가 해야 할 일:
- 아직 하면 안 되는 일:

기획 정리:
1. 이 기능이 해결하는 문제
2. 핵심 사용자
3. 사용자 시나리오
4. MVP 포함 여부
5. 필수 기능 / 나중 기능 / 제외 기능
6. 성공 기준
7. 다음 설계 단계에서 결정해야 할 질문
```

---

## 5. 기획자 체크리스트

기획자는 아래 질문에 답할 수 있어야 한다.

```text
이 기능은 왜 필요한가?
이 기능이 없으면 핵심 서비스가 안 되는가?
지금 꼭 해야 하는가?
나중에 해도 되는가?
이번 작업에서 일부러 하지 않을 것은 무엇인가?
사용자가 성공했다고 느끼는 순간은 언제인가?
```
