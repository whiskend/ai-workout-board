# 04. Design Guide

이 문서는 설계자 역할을 수행할 때 사용하는 문서다.

설계 단계의 목표는 “어떤 구조로 만들지”를 결정하는 것이다.

---

## 1. 설계 단계에서 코드 수정 금지

설계 단계에서는 코드를 수정하지 않는다.

먼저 아래를 정한다.

```text
- 이 기능은 어느 영역의 책임인가?
- 새 API가 필요한가?
- DB 변경이 필요한가?
- 화면은 어떻게 바뀌는가?
- 외부 서비스 호출이 필요한가?
- 어떤 파일이 수정될 가능성이 있는가?
```

---

## 2. 설계자가 결정할 것

```text
1. A안/B안/C안 중 무엇으로 갈지
2. DB를 변경할지 말지
3. 새 API를 만들지 기존 API를 쓸지
4. 결과를 저장할지 화면에만 보여줄지
5. 어느 계층/모듈이 어떤 책임을 가질지
6. 이번 작업 범위를 어디까지로 자를지
```

---

## 3. 책임 분리 예시

프로젝트 기술 스택과 관계없이 아래처럼 역할을 나눠 생각한다.

```text
UI / Client:
- 사용자의 입력을 받는다.
- 결과를 화면에 표시한다.
- 직접 DB를 만지지 않는다.

API / Backend:
- 요청을 받는다.
- 인증과 권한을 확인한다.
- 필요한 데이터를 조회하거나 저장한다.
- 외부 서비스 호출을 조율한다.

Service / Domain Logic:
- 실제 비즈니스 규칙을 처리한다.
- 어떤 데이터를 어떻게 사용할지 판단한다.

Repository / Data Access:
- DB 조회와 저장을 담당한다.
- 비즈니스 판단을 많이 넣지 않는다.

External / AI / Tool:
- 외부 API 호출 또는 분석 작업을 담당한다.
- 핵심 DB 권한 판단을 대신하지 않는다.
```

현재 프로젝트에 Repository 계층이 없다면 억지로 만들 필요는 없다.
다만 DB 접근 코드가 너무 커지면 나중에 분리 후보로 기록한다.

## 5. 설계 단계 Codex 프롬프트

```text
AGENTS.md와 docs/04-design-guide.md를 읽어줘.
아직 코드는 수정하지 마.

지금은 설계 단계야.
이번 기능은 [기능 이름]이야.

아래 형식으로 먼저 설명해줘.

역할 확인:
- 현재 단계:
- 사용자의 역할:
- Codex의 역할:
- 지금 사용자가 해야 할 일:
- 아직 하면 안 되는 일:

설계 정리:
1. 이 기능의 목표
2. 이 기능을 만들기 전에 내가 결정해야 할 질문
3. 가능한 설계안 2~3개
4. 각 설계안의 장단점
5. 지금 프로젝트 수준에서 추천하는 안
6. 추천 이유
7. 수정될 가능성이 있는 파일
8. 변경하지 말아야 할 영역
9. API 영향
10. DB 영향
11. 테스트 전략
12. 초보자인 내가 이해해야 할 핵심
```

## 6. 설계자 체크리스트

```text
이 기능은 어떤 사용자 행동에서 시작되는가?
어느 화면/API/DB가 관여하는가?
이 데이터는 저장해야 하는가, 즉시 보여주기만 하면 되는가?
인증이나 권한 검사가 필요한가?
새 API가 필요한가?
기존 API 응답 형식을 깨뜨리지는 않는가?
DB schema 변경이 정말 필요한가?
이번 작업이 너무 커지지는 않았는가?
나중에 확장할 수 있는가?
```
