## 2026-06-15 - AI 분석 결과 저장 방식

결정:
- AI 분석 결과는 DB에 저장하지 않고, 요청할 때마다 화면에만 보여준다.

선택지:
- A안: DB에 저장하지 않고 화면에만 표시
- B안: DB에 저장해서 분석 이력 제공

선택한 이유:
- 현재 목표는 AI 분석 흐름을 데모 가능한 수준으로 연결하는 것이다.
- DB schema 변경과 migration을 피할 수 있다.
- 분석 이력 저장보다 현재 기록과 이전 기록을 비교해 화면에 보여주는 것이 발표 데모에 더 중요하다.
- 분석 결과 저장까지 넣으면 `AnalysisResult` 같은 새 테이블 설계가 필요해지고 작업 범위가 커진다.

포기한 것:
- 사용자가 과거 AI 분석 결과를 다시 조회하는 기능
- 분석 결과 히스토리 관리

나중에 다시 볼 것:
- 사용자가 분석 결과를 저장하고 싶어 할 때 `AnalysisResult` 테이블 추가 검토
- 분석 결과 품질 평가나 피드백 기능이 필요할 때 저장 방식 재검토

## 2026-06-15 - React와 FastAPI 통신 방식

결정:
- React는 FastAPI를 직접 호출하지 않는다.
- AI 분석 요청은 반드시 NestJS를 거친다.

선택지:
- A안: React -> NestJS -> FastAPI
- B안: React -> FastAPI 직접 호출

선택한 이유:
- React가 FastAPI를 직접 호출하면 인증과 권한 검사를 프론트 또는 FastAPI 쪽에서 다시 고민해야 한다.
- 현재 프로젝트에서 JWT 검증과 작성자 권한 검사는 NestJS가 담당한다.
- NestJS가 현재 게시글과 이전 운동 기록을 PostgreSQL에서 조회한 뒤 FastAPI에 전달하는 구조가 책임 분리에 맞다.
- FastAPI는 AI 분석 처리에 집중하고, 사용자 인증이나 DB 접근 책임은 갖지 않는 것이 안전하다.

포기한 것:
- React에서 FastAPI를 바로 호출하는 단순한 연결 방식
- FastAPI가 직접 DB를 조회하는 구조

나중에 다시 볼 것:
- AI 서버가 별도 인증 체계를 가져야 할 정도로 커지는 경우 내부 API 인증 방식 강화
- 배포 환경에서 NestJS와 FastAPI 사이 네트워크 접근 제한 방식 검토

## 2026-06-15 - AI 분석과 RAG 최소 구현 작업 범위

결정:
- #9 AI 운동 기록 분석 기능과 #10 RAG 기능을 같은 브랜치에서 함께 진행한다.

선택지:
- A안: #9 AI 분석과 #10 RAG 최소 구현을 함께 진행
- B안: #9 AI 분석만 먼저 하고, #10 RAG는 별도 브랜치에서 진행

선택한 이유:
- 이 프로젝트의 AI 분석은 이전 운동 기록을 참고해야 의미가 있다.
- 이전 기록 조회 없이 AI 분석만 붙이면 운동 기록 서비스의 핵심인 점진적 과부하 추천이 약해진다.
- 현재 RAG는 pgvector까지 가는 완성형 검색이 아니라, 같은 사용자와 같은 운동명의 과거 기록을 조회하는 구조화 검색 기반 최소 구현이다.
- DB schema 변경 없이 `Post`, `Exercise`, `ExerciseSet` 기존 데이터만으로 구현할 수 있다.

포기한 것:
- pgvector 기반 semantic search
- 여러 운동명을 의미적으로 묶는 고급 검색
- 분석 결과 DB 저장

나중에 다시 볼 것:
- 운동명 정규화 tool이 들어간 뒤 검색 품질 개선
- pgvector를 이용한 유사 운동 기록 검색
- 분석 결과 저장이 필요한지 재검토

## 2026-06-15 - OpenAI 호출 실패 시 fallback

결정:
- FastAPI는 `OPENAI_API_KEY`가 있으면 OpenAI Responses API 호출을 먼저 시도한다.
- API 키가 없거나 호출이 실패하면 rule-based 분석 결과를 반환한다.

선택지:
- A안: OpenAI 호출 실패 시 rule-based 분석으로 fallback
- B안: OpenAI 호출 실패 시 에러를 그대로 사용자에게 노출

선택한 이유:
- 발표와 데모 중 API 키 누락, 네트워크 문제, 요금 문제로 전체 흐름이 끊기면 위험하다.
- rule-based 분석만으로도 이전 기록 대비 반복 수 증가, 목표 반복 수 달성 여부, 다음 목표 추천을 설명할 수 있다.
- OpenAI 연결은 품질 향상 수단이고, 서비스의 최소 흐름은 fallback으로 유지할 수 있다.

포기한 것:
- OpenAI 실패를 사용자에게 즉시 드러내는 단순한 구조
- 항상 LLM 응답만 보여주는 구조

나중에 다시 볼 것:
- OpenAI 응답 JSON 검증 강화
- fallback이 발생했는지 운영 로그로 남기는 방식
- 사용자 화면에 분석 모드를 얼마나 노출할지 결정
