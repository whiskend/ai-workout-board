# 05. Review Guide

이 문서는 리뷰어 역할을 수행할 때 사용하는 문서다.

리뷰 단계의 목표는 Codex가 만든 코드가 **우리가 정한 설계와 맞는지** 확인하는 것이다.

---

## 1. 리뷰 단계에서 기본적으로 코드 수정 금지

리뷰 단계에서는 먼저 검토만 한다.

수정이 필요하면 다음 순서로 진행한다.

```text
1. 문제를 찾는다.
2. 위험도를 나눈다.
3. 수정 후보를 설명한다.
4. 사용자가 승인한다.
5. 그때 수정한다.
```

---

## 2. 리뷰어가 보는 핵심 4가지

```text
1. 우리가 정한 설계를 지켰는가?
2. 보안/권한 문제가 없는가?
3. 기존 기능을 깨뜨리지 않았는가?
4. 테스트로 확인 가능한가?
```

---

## 3. 변경 파일 리뷰

Codex에게 먼저 파일별 요약을 받는다.

```text
변경된 파일:
- file A: 어떤 역할 때문에 변경됨
- file B: 어떤 역할 때문에 변경됨
```

그다음 중요한 파일부터 diff를 확인한다.

---

## 4. 리뷰 위험도 분류

| 위험도 | 의미 | 예시 |
|---|---|---|
| 높음 | 지금 고치지 않으면 기능/보안이 깨짐 | 남의 데이터 접근 가능, 인증 누락 |
| 중간 | 곧 고치는 게 좋음 | 날짜 조건 누락, 에러 처리 부족 |
| 낮음 | 나중에 개선 가능 | UI 문구, 중복 코드 약간 |

---

## 5. 일반 리뷰 체크리스트

```text
설계:
- 이번 변경이 사용자가 승인한 범위 안에 있는가?
- 새 기능이 기존 구조를 불필요하게 흔들지 않았는가?
- 한 번에 너무 많은 기능을 섞지 않았는가?

API:
- 필요한 API에 인증이 걸려 있는가?
- 권한 검사가 필요한 경우 실제로 검사하는가?
- 기존 API 응답 형식을 깨뜨리지 않았는가?

DB:
- DB schema를 불필요하게 바꾸지 않았는가?
- 현재 작업에 필요 없는 migration이 생기지 않았는가?
- 내 데이터와 남의 데이터가 섞이지 않는가?

Frontend:
- 화면이 직접 DB나 민감한 외부 서비스를 호출하지 않는가?
- 로딩/에러 상태가 있는가?
- 사용자가 실패 상황을 이해할 수 있는가?

External / AI:
- 외부 API 실패 시 전체 화면이 깨지지 않는가?
- 민감한 값을 프론트에 노출하지 않는가?
- 결과를 저장해야 하는지/저장하지 말아야 하는지 설계와 맞는가?

문서:
- 문서가 현재 코드 상태와 맞는가?
- 과장된 표현은 없는가?
```

---

## 6. 리뷰 단계 Codex 프롬프트

```text
AGENTS.md와 docs/05-review-guide.md를 읽어줘.
아직 코드는 수정하지 마.

지금은 리뷰 단계야.
현재 변경사항을 기준으로 아래를 확인해줘.

역할 확인:
- 현재 단계:
- 사용자의 역할:
- Codex의 역할:
- 지금 사용자가 해야 할 일:
- 아직 하면 안 되는 일:

리뷰 결과:
1. 변경된 파일별 역할 요약
2. 설계상 맞는 부분
3. 위험한 부분
4. 지금 당장 고쳐야 하는 부분
5. 나중에 개선해도 되는 부분
6. 사용자가 직접 확인해야 할 부분
7. 테스트해야 할 시나리오
8. 커밋해도 되는 상태인지 판단

주의:
- 문제를 발견해도 바로 수정하지 마.
- 먼저 리뷰 결과만 보고해.
```

---

## 7. 리뷰어의 최종 판단 문장

리뷰가 끝나면 사용자는 아래 중 하나로 결정한다.

```text
- 커밋해도 된다.
- 작은 수정 후 커밋하자.
- 지금은 위험하니 되돌리자.
- 구현 범위를 줄여서 다시 하자.
```
